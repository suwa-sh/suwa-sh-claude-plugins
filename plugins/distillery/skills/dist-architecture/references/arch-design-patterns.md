## 回復性パターン

### Circuit Breaker

| 項目 | 内容 |
|------|------|
| 概要 | リモートサービスの障害が継続する場合に接続を遮断し、障害の連鎖（カスケード障害）を防止する。Closed（正常通過）→ Open（遮断）→ Half-Open（試行）の3状態で動作する |
| RDRA シグナル | 外部システム.tsv にシステムが存在する（特に決済・認証等の重要連携）。BUC に外部システム呼び出しを含むアクティビティが存在 |
| NFR シグナル | NFR A.1.2（サービス切替時間）、NFR A.2.1（サーバ冗長化）。可用性グレードが高いほど適用優先度が上がる |
| ティアへの影響 | 外部連携ティア、Backend API の Infrastructure 層 |
| 方針/ルールへの影響 | ティアの policy として「サーキットブレーカー適用」を追加。Open 状態での代替応答（フォールバック）方針を rule に含める |
| トレードオフ | 障害伝播防止・システム安定性向上 ↔ 実装複雑性・Half-Open 閾値チューニング・フォールバック応答の設計 |
| 適用すべきでない場合 | 呼び出し先がインメモリリソースやローカルキャッシュの場合。レイテンシが問題にならないバッチ処理で単純リトライが十分な場合 |

### Retry

| 項目 | 内容 |
|------|------|
| 概要 | 一時的な障害（ネットワーク瞬断、サービス一時過負荷等）に対し、自動的に操作を再試行する。指数バックオフ（待機時間を段階的に増加）と Jitter（ランダムな揺らぎ）を組み合わせて、再試行集中（Thundering Herd）を回避する |
| RDRA シグナル | 外部システム連携がある BUC。特にネットワーク越しの呼び出しを伴うアクティビティ |
| NFR シグナル | NFR A.1.2（サービス切替時間）、NFR A.2.1（サーバ冗長化） |
| ティアへの影響 | 外部連携ティア、Backend API の Infrastructure 層 |
| 方針/ルールへの影響 | 外部連携の rule として「指数バックオフ + Jitter による再試行」を追加。最大リトライ回数と総タイムアウトを policy に含める |
| トレードオフ | 一時障害への自動回復 ↔ レイテンシ増加・冪等でない操作での副作用リスク（重複処理） |
| 適用すべきでない場合 | 恒久的な障害（認証エラー、権限不足、リソース未存在等の 4xx 系エラー）。冪等でない操作で副作用の重複が許容できない場合 |

### Bulkhead

| 項目 | 内容 |
|------|------|
| 概要 | コンポーネントをリソースプール（スレッドプール、コネクションプール等）に隔離し、1つの障害が他に波及するのを防ぐ。船舶の隔壁に由来する概念 |
| RDRA シグナル | 外部システムが3つ以上、または重要度の異なる業務が混在する BUC 構成 |
| NFR シグナル | NFR A.2.1（サーバ冗長化）>= Lv3（高可用性要件） |
| ティアへの影響 | Backend API のコネクションプール設計、外部連携ティアのリソース分離 |
| 方針/ルールへの影響 | ティア共通ルールとして「接続プール分離」を追加。外部システムごと・業務重要度ごとにプールを分離する方針を policy に含める |
| トレードオフ | 障害分離・影響範囲の限定 ↔ リソース利用効率の低下（予約済みリソースの余剰） |
| 適用すべきでない場合 | リソース隔離コストがサービス重要度に見合わない場合。外部連携先が1つのみで障害分離の意味がない場合 |

### Health Endpoint Monitoring

| 項目 | 内容 |
|------|------|
| 概要 | 外部ツールからポーリングできるヘルスチェックエンドポイントを公開し、アプリケーションやサービスの正常性を継続的に監視する。依存サービス（DB、MQ、外部 API 等）の接続状態も含めた深度のあるチェックを提供する |
| RDRA シグナル | （常に推奨。ティアが2つ以上の場合は特に重要） |
| NFR シグナル | NFR A.2.1（サーバ冗長化）>= Lv3、NFR C.1.3（運用監視） |
| ティアへの影響 | 全ティアにヘルスチェックエンドポイントを追加。LB のバックエンド振り分け判定にも使用 |
| 方針/ルールへの影響 | ティア共通方針として「ヘルスチェック」を追加。チェック深度（shallow: プロセス生存確認 / deep: 依存サービス確認）を policy に含める |
| トレードオフ | 障害検知の即時性・自動復旧との連携 ↔ ヘルスチェック自体の負荷・深度チェックによる依存サービスへの負荷 |
| 適用すべきでない場合 | （基本的に常に適用すべき。単一プロセスのバッチジョブでは簡易的なプロセス監視で代替可能） |

### Timeout

| 項目 | 内容 |
|------|------|
| 概要 | リモート呼び出しの応答を待つ上限時間を設定し、リソース（スレッド、コネクション）の無限占有を防ぐ。Circuit Breaker や Retry と組み合わせて回復性を高める |
| RDRA シグナル | 外部システム連携がある BUC。レスポンスタイムが重要な画面操作系 UC |
| NFR シグナル | NFR B.2.1（レスポンスタイム目標）。目標レスポンスタイムからタイムアウト値の上限を逆算 |
| ティアへの影響 | 外部連携ティア、Backend API、API Gateway |
| 方針/ルールへの影響 | 外部連携 rule として「タイムアウト設定」を追加。ティアごとにタイムアウト値の階層を定義（Gateway > Backend API > 外部連携） |
| トレードオフ | リソース解放・障害検知の迅速化 ↔ 短すぎるタイムアウトによる正常リクエストの切断リスク |
| 適用すべきでない場合 | 処理時間が本質的に長い操作（大量データ集計等）では Asynchronous Request-Reply パターンに切り替える |

### Rate Limiting / Throttling

| 項目 | 内容 |
|------|------|
| 概要 | 単位時間あたりのリクエスト数を制限し、サービスの過負荷を防止する。Token Bucket、Sliding Window 等のアルゴリズムで実現する。クライアント単位・API 単位・グローバル単位でレート制限を適用できる |
| RDRA シグナル | 外部アクターが存在する（公開 Web サービス）。外部システムからの API 呼び出しを受ける |
| NFR シグナル | NFR B.1.1（同時アクセス数）、NFR B.1.2（ピーク時業務量）。NFR E.10（不正アクセス対策）にも関連 |
| ティアへの影響 | API Gateway ティア。API Gateway がない場合は Backend API のミドルウェア層 |
| 方針/ルールへの影響 | API Gateway の policy として「レート制限」を追加。制限超過時の応答（429 Too Many Requests + Retry-After ヘッダ）を rule に含める |
| トレードオフ | 過負荷防止・公平なリソース配分 ↔ 正当なバーストトラフィックの抑制リスク |
| 適用すべきでない場合 | 内部サービス間の通信のみで外部公開がない場合（ただし内部でも保護が必要なケースあり） |

### Saga

| 項目 | 内容 |
|------|------|
| 概要 | 複数サービスをまたぐ長時間トランザクションを、ローカルトランザクションの連鎖として管理する。各ステップに補償トランザクション（Compensating Transaction）を定義し、途中で失敗した場合に完了済みステップを巻き戻す。オーケストレーション方式（中央コーディネーター）とコレオグラフィー方式（イベント連鎖）がある |
| RDRA シグナル | 状態遷移が外部システム連携を含む（例: 精算→決済機関）。複数エンティティにまたがる状態遷移。BUC に「予約→決済→完了」のような複数ステップのビジネスフローが存在 |
| NFR シグナル | NFR A.4.1（RPO/RTO）。データ整合性と可用性の両立が必要な場合 |
| ティアへの影響 | Backend API の UseCase 層（オーケストレーション方式）、Worker ティア（コレオグラフィー方式）、MQ ティア |
| 方針/ルールへの影響 | UseCase 層の policy として「分散トランザクション方式: Saga」を追加。各ステップの補償トランザクションを rule に必ず含める |
| トレードオフ | 分散環境での整合性確保・サービス自律性 ↔ 補償トランザクション設計の複雑性・最終的一貫性（結果整合性）の許容が必要 |
| 適用すべきでない場合 | 単一サービス内でトランザクションが完結する場合。ACID トランザクションで十分な場合 |

### Compensating Transaction

| 項目 | 内容 |
|------|------|
| 概要 | 完了済みの操作の効果を取り消す一連の操作。Saga パターンの構成要素であり、分散システムで結果整合性を実現する際の障害回復メカニズム |
| RDRA シグナル | Saga パターン適用時に自動的に必要。複数外部システムにまたがる業務フロー |
| NFR シグナル | NFR A.4.1（RPO/RTO）、NFR A.1.2（サービス切替時間） |
| ティアへの影響 | Backend API の UseCase 層、Worker ティア |
| 方針/ルールへの影響 | Saga の各ステップに対応する補償操作を rule として定義。補償操作自体の冪等性も rule に含める |
| トレードオフ | 障害時のデータ整合性回復 ↔ 補償ロジックの設計・テスト・運用コスト |
| 適用すべきでない場合 | 操作が本質的に取り消し不可能な場合（物理的なアクション等）。この場合は人手介入フローを設計する |

### Queue-Based Load Leveling

| 項目 | 内容 |
|------|------|
| 概要 | リクエスト発行側と処理側の間にキュー（MQ）をバッファとして配置し、需要の急増を吸収して処理の平準化を図る。プロデューサーはキューに投入するだけで完了し、コンシューマーは自身のペースで処理する |
| RDRA シグナル | BUC に「バッチ」「一括」「通知」「非同期」「集計」を含むアクティビティ。ピーク時に処理遅延が許容される業務 |
| NFR シグナル | NFR B.1.2（ピーク時業務量）、NFR B.2.2（バッチ処理時間） |
| ティアへの影響 | MQ ティア + Worker ティア追加。Backend API は MQ へのプロデューサーとなる |
| 方針/ルールへの影響 | ワーカーティアの policy として「リトライ・DLQ（Dead Letter Queue）」を追加。DLQ のモニタリングと手動再処理方針を rule に含める |
| トレードオフ | ピーク負荷の吸収・処理の安定化 ↔ 処理遅延の発生（リアルタイム性の低下）・MQ 運用コスト |
| 適用すべきでない場合 | リアルタイムレスポンスが必須で遅延が許容できない場合 |

## データ管理パターン

### Cache-Aside

| 項目 | 内容 |
|------|------|
| 概要 | アプリケーションがキャッシュとデータストアを個別に管理する。読み取り時にまずキャッシュを確認し、キャッシュミスの場合にデータストアから取得してキャッシュに格納する（Lazy Loading）。書き込み時はデータストアを更新し、キャッシュを無効化する |
| RDRA シグナル | 情報.tsv に検索頻度の高いマスタデータ（会議室情報、カテゴリ情報等）。BUC に検索・照会系 UC が多い |
| NFR シグナル | NFR B.2.1（レスポンスタイム目標）。読み取り負荷が書き込み負荷より大きい場合に効果的 |
| ティアへの影響 | KVS ティア追加、storage_mapping に cache を追加（RDB との二重マッピング） |
| 方針/ルールへの影響 | KVS ティアの policy として「TTL 管理」を追加。キャッシュ無効化戦略（TTL ベース / イベントベース）を rule に含める |
| トレードオフ | 読み取り性能の大幅改善・DB 負荷軽減 ↔ データ鮮度の低下（TTL 期間中の古いデータ）・キャッシュ invalidation の複雑性 |
| 適用すべきでない場合 | データが頻繁に更新され、常に最新値が必要な場合。キャッシュ対象データが大きすぎる場合 |

### CQRS（Command Query Responsibility Segregation）

| 項目 | 内容 |
|------|------|
| 概要 | 読み取り（Query）と書き込み（Command）のモデルを分離する。それぞれ独立したデータストアを持つことも可能で、読み書きを独立にスケールできる。Query 側にはマテリアライズドビュー（Materialized View）を用いて読み取り最適化を行うことが多い |
| RDRA シグナル | BUC に検索・照会系 UC と登録・更新系 UC が混在し、読み書き負荷が非対称。情報.tsv のエンティティが多く、複雑な検索条件がある |
| NFR シグナル | NFR B.1.1（同時アクセス数）>= Lv3（高同時アクセス数）、NFR B.2.1（レスポンス要件） |
| ティアへの影響 | Backend API のレイヤー構造（Command/Query の UseCase 分離）。必要に応じて Query 専用データストア追加 |
| 方針/ルールへの影響 | アプリケーションアーキテクチャの方針として「読み書き分離」を追加。Command → Query の同期方式（同期/非同期）を rule に含める |
| トレードオフ | 読み書きの独立スケール・各モデルの最適化 ↔ 最終的一貫性（Command から Query への反映遅延）・モデル二重管理のコスト |
| 適用すべきでない場合 | 読み書き負荷が対称で、単純な CRUD で要件を満たせる場合。データモデルが単純で分離の恩恵が薄い場合 |

### Event Sourcing

| 項目 | 内容 |
|------|------|
| 概要 | エンティティの現在状態ではなく、状態変更をイベントとして追記専用ストアに記録する。任意の時点の状態をイベントのリプレイで復元できる。監査証跡の完全性、時系列分析、デバッグ容易性を提供する |
| RDRA シグナル | 状態.tsv の状態遷移が多く（8種以上）、変更履歴の完全な追跡が必要。条件.tsv に監査・証跡に関するビジネスルールがある |
| NFR シグナル | NFR E.7.1（監査ログ）、NFR C.6.1（ログ保管）。金融・医療等の規制対応で完全な変更履歴が求められる場合 |
| ティアへの影響 | データストアティア（イベントストア追加）、Worker ティア（プロジェクション用イベントハンドラ） |
| 方針/ルールへの影響 | データアーキテクチャの方針として「イベントソーシング」を追加。スナップショット戦略（N イベントごと）、イベントスキーマ進化方針を rule に含める |
| トレードオフ | 完全な変更履歴・時点復元・監査対応 ↔ リハイドレート（リプレイ）コスト・スキーマ進化の複雑性・アーキテクチャ全体への影響が大きい |
| 適用すべきでない場合 | イベント履歴が不要で現在状態のみ管理すれば十分な場合。単純な CRUD アプリケーション |

### Materialized View

| 項目 | 内容 |
|------|------|
| 概要 | 複数のデータソースから事前集計・変換済みのビューを生成し、クエリ性能を最適化する。CQRS の Query 側データストアとして、または複雑な集計レポートの事前計算に使用する |
| RDRA シグナル | BUC にダッシュボード・レポート・集計表示系の UC がある。情報.tsv のエンティティを跨いだ結合クエリが頻繁 |
| NFR シグナル | NFR B.2.1（レスポンスタイム目標）。集計・レポート系の表示速度が重要 |
| ティアへの影響 | データストアティア（Read 用ストア追加）、Worker ティア（ビュー更新処理） |
| 方針/ルールへの影響 | データアーキテクチャの rule としてビュー更新トリガー（イベント駆動 / 定期バッチ）を定義 |
| トレードオフ | クエリ性能の大幅改善 ↔ データ鮮度・ストレージコスト・ビュー更新処理の運用 |
| 適用すべきでない場合 | ソースデータが頻繁に変わり、ビュー更新コストが高い場合。リアルタイム性が最優先の場合 |

### Sharding

| 項目 | 内容 |
|------|------|
| 概要 | データストアを水平パーティション（シャード）に分割し、各シャードが独立してスケールする。シャードキーの選定が性能と運用の鍵となる |
| RDRA シグナル | 情報.tsv のエンティティ数が多く、単一 DB の限界が予見される。マルチテナント構成で、テナントごとのデータ分離が必要 |
| NFR シグナル | NFR B.1.1（データ量）>= Lv3、NFR B.3.3（ストレージ拡張性） |
| ティアへの影響 | データストアティア（RDB シャーディング構成） |
| 方針/ルールへの影響 | データアーキテクチャの rule としてシャードキー選定基準、リバランシング方針を定義 |
| トレードオフ | 水平スケーラビリティ・データ分離 ↔ クロスシャードクエリの複雑性・シャードキー変更の困難さ |
| 適用すべきでない場合 | データ量が少なく、単一 DB で十分にスケールできる場合。クロスシャードの結合クエリが頻繁に必要な場合 |

### Static Content Hosting

| 項目 | 内容 |
|------|------|
| 概要 | 静的コンテンツ（HTML、CSS、JS、画像等）をオブジェクトストレージや CDN から直接配信し、コンピュートリソースの負荷を軽減する |
| RDRA シグナル | 情報.tsv に画像・ファイル属性を持つエンティティがある。フロントエンドティアが SPA の場合 |
| NFR シグナル | NFR B.2.1（レスポンスタイム目標）、NFR B.1.2（ピーク時の配信性能） |
| ティアへの影響 | CDN ティア追加、Object Storage ティア |
| 方針/ルールへの影響 | CDN のキャッシュ制御方針（Cache-Control ヘッダ）を rule に含める |
| トレードオフ | 配信性能・コンピュートコスト削減 ↔ キャッシュ無効化の即時性 |
| 適用すべきでない場合 | 静的コンテンツがほぼない場合 |

## メッセージングパターン

### Publisher-Subscriber

| 項目 | 内容 |
|------|------|
| 概要 | メッセージの送信者（パブリッシャー）と受信者（サブスクライバー）を疎結合にする非同期イベント通知。パブリッシャーはサブスクライバーを意識せずイベントを発行し、メッセージブローカーがルーティングを担当する。1つのイベントを複数のサブスクライバーが受信できる |
| RDRA シグナル | BUC に「通知」「メール送信」「連携」を含むアクティビティ。複数業務が同一イベントを消費する構造（例: 予約確定→通知送信 + 在庫更新） |
| NFR シグナル | NFR B.1.2（ピーク時対応）。サービス間の疎結合性が求められる場合 |
| ティアへの影響 | MQ ティア、Worker ティア（サブスクライバー） |
| 方針/ルールへの影響 | MQ ティアの policy として「メッセージ順序保証方針」「重複配信への対応（冪等消費）」を追加 |
| トレードオフ | サービス間の疎結合・イベント駆動アーキテクチャ ↔ メッセージ順序保証の難しさ・デバッグの複雑さ |
| 適用すべきでない場合 | 同期的なリクエスト/レスポンスが必要な場合。サブスクライバーが常に1つだけの場合（Point-to-Point で十分） |

### Competing Consumers

| 項目 | 内容 |
|------|------|
| 概要 | 単一のキューに対して複数のコンシューマーが並行してメッセージを処理する。各メッセージは1つのコンシューマーのみが処理する（排他的消費）。コンシューマー数の増減でスループットを水平スケールする |
| RDRA シグナル | BUC にバッチ/一括処理があり、処理量が大きい。通知送信等の大量メッセージ処理が必要 |
| NFR シグナル | NFR B.2.2（バッチ処理時間）、NFR B.3.1（CPU 拡張性） |
| ティアへの影響 | Worker ティアのスケーリング方針（水平スケール） |
| 方針/ルールへの影響 | Worker ティアの policy として「オートスケーリング方針」「メッセージ処理の冪等性」を追加 |
| トレードオフ | スループットの水平スケール ↔ メッセージ順序保証の喪失・コンシューマー間のリソース競合 |
| 適用すべきでない場合 | メッセージの厳密な順序保証が必要な場合 |

### Asynchronous Request-Reply

| 項目 | 内容 |
|------|------|
| 概要 | 長時間かかる処理をバックグラウンドで実行し、クライアントにはリクエスト受付の即時応答（202 Accepted）を返す。クライアントはポーリングまたはコールバックで処理結果を取得する |
| RDRA シグナル | BUC に処理時間が長いアクティビティ（精算額計算、データ集計、レポート生成、大量データインポート等） |
| NFR シグナル | NFR B.2.1（レスポンスタイム）、NFR B.2.2（バッチ処理時間）。ユーザー体験としてレスポンスの即時性が求められる場合 |
| ティアへの影響 | Backend API（受付 + ステータス照会）+ Worker ティア（バックグラウンド処理）の連携 |
| 方針/ルールへの影響 | Backend API の rule として「ジョブステータス管理 API」を追加。Worker ティアの rule として「進捗更新」を追加 |
| トレードオフ | ユーザー体験の即時性・長時間処理のスケーラビリティ ↔ ステータス管理の複雑性・ポーリングの負荷 |
| 適用すべきでない場合 | 処理が十分高速で同期レスポンスが可能な場合 |

### Claim Check

| 項目 | 内容 |
|------|------|
| 概要 | 大きなメッセージ本体を外部ストレージに格納し、メッセージにはその参照（Claim Check）のみを含める。メッセージブローカーの負荷を軽減し、大量データの効率的な受け渡しを実現する |
| RDRA シグナル | BUC にファイルアップロード・大量データ連携を含むアクティビティ。情報.tsv に画像・ファイル属性を持つエンティティがありメッセージングと組み合わせる場合 |
| NFR シグナル | NFR B.1.2（ピーク時業務量）。MQ のメッセージサイズ制限に抵触する場合 |
| ティアへの影響 | MQ ティア + Object Storage ティア |
| 方針/ルールへの影響 | MQ の rule として「メッセージサイズ上限」を定義。上限超過時の Claim Check ルーティングを含める |
| トレードオフ | MQ 負荷軽減・大きなデータの受け渡し ↔ 2段階の取得処理による実装複雑性 |
| 適用すべきでない場合 | メッセージが常に小さく、MQ のサイズ制限内に収まる場合 |

### Priority Queue

| 項目 | 内容 |
|------|------|
| 概要 | メッセージに優先度を付与し、優先度の高いメッセージを先に処理する。SLA が異なるクライアント種別や、業務重要度の異なるリクエストを適切に振り分ける |
| RDRA シグナル | アクター種別が複数あり、サービスレベルが異なる。BUC に優先度の異なる業務フローが混在 |
| NFR シグナル | NFR B.2.1（レスポンスタイム）でアクター種別ごとに異なる要件がある場合 |
| ティアへの影響 | MQ ティア（優先度付きキュー）、Worker ティア |
| 方針/ルールへの影響 | MQ の policy として「優先度分類基準」を定義 |
| トレードオフ | 重要な処理の優先実行 ↔ 低優先度メッセージの枯渇（Starvation）リスク |
| 適用すべきでない場合 | 全メッセージが同一優先度で処理される場合 |

## セキュリティパターン

### Federated Identity

| 項目 | 内容 |
|------|------|
| 概要 | 認証処理を外部 IdP（Identity Provider）に委譲する。アプリケーションは認証ロジックを持たず、IdP が発行したトークン（OAuth2/OIDC）を検証する。ユーザー管理・パスワードポリシー・MFA 等を IdP に集約できる |
| RDRA シグナル | 外部アクター（社外ユーザー）が存在する。複数アプリケーションでの SSO が必要 |
| NFR シグナル | NFR E.5.1（認証方式）。セキュリティ要件が高い場合 |
| ティアへの影響 | IdP ティア追加。API Gateway でのトークン検証 |
| 方針/ルールへの影響 | ティア共通方針として「認証方式: OAuth2/OIDC」を追加。トークンライフサイクル管理（アクセストークン有効期限、リフレッシュトークン方針）を rule に含める |
| トレードオフ | 認証の一元管理・標準プロトコル準拠・セキュリティ強化 ↔ IdP への依存・IdP 障害時の全面影響 |
| 適用すべきでない場合 | 内部ユーザーのみで簡易認証が十分な場合 |

### Valet Key

| 項目 | 内容 |
|------|------|
| 概要 | クライアントに対し、特定リソース（ファイル、オブジェクト等）への制限付き直接アクセスを可能にするトークン（署名付き URL 等）を発行する。Backend API を経由せずに大容量データの直接アップロード/ダウンロードが可能になり、スループットを改善する |
| RDRA シグナル | 情報.tsv に画像・ファイル属性を持つエンティティがある。BUC にファイルアップロード/ダウンロードのアクティビティがある |
| NFR シグナル | NFR E.6.1（データ暗号化）、NFR B.2.1（レスポンスタイム）。大容量ファイルの転送が頻繁な場合 |
| ティアへの影響 | Object Storage ティア。Backend API はトークン発行のみ担当 |
| 方針/ルールへの影響 | Object Storage の policy として「署名付き URL」を追加。URL の有効期限・アクセス権限範囲を rule に含める |
| トレードオフ | Backend API の負荷軽減・大容量転送のスループット向上 ↔ トークン漏洩リスク・有効期限管理の複雑性 |
| 適用すべきでない場合 | ファイルが小さく Backend API 経由でも性能問題がない場合 |

### Gatekeeper / Gateway Offloading

| 項目 | 内容 |
|------|------|
| 概要 | 共通のセキュリティ機能（認証、認可、TLS 終端、レート制限、WAF、IP 制限等）をゲートウェイに集約し、各バックエンドサービスから横断的関心事をオフロードする。バックエンドサービスはビジネスロジックに集中できる |
| RDRA シグナル | 外部アクター2種以上。フロントエンド2種以上（利用者向け + 管理者向け等）。外部システムからの API 呼び出しを受ける |
| NFR シグナル | NFR E.5.1（認証）、NFR E.5.2（アクセス制御）、NFR E.10（WAF）。セキュリティ対策を一元化する要件 |
| ティアへの影響 | API Gateway ティア追加。TLS 終端・トークン検証・RBAC の粗粒度チェックを Gateway で実行 |
| 方針/ルールへの影響 | API Gateway の policy として「セキュリティオフロード」を追加。Gateway と Backend API のセキュリティ責務分界を rule に含める |
| トレードオフ | セキュリティ一元管理・バックエンド簡素化 ↔ Gateway の単一障害点化リスク・Gateway 設定の複雑性 |
| 適用すべきでない場合 | Backend API が1つのみで Gateway を導入するまでもない場合 |

## 設計・構造パターン

### Backends for Frontends（BFF）

| 項目 | 内容 |
|------|------|
| 概要 | フロントエンド種別（モバイル、デスクトップ、管理画面等）ごとに専用の Backend API を作成する。各 BFF はそのフロントエンドに最適化されたデータ集約・変換を担当し、汎用 API の過剰/不足問題を解消する |
| RDRA シグナル | アクター種別が3以上で UI 要件が大きく異なる（モバイル vs デスクトップ vs 管理画面）。BUC の UC に画面固有の集約パターンがある |
| NFR シグナル | NFR F.1.1（対応デバイス）で多様なデバイス対応が求められる場合 |
| ティアへの影響 | Backend API ティアをフロントエンド種別ごとに分割 |
| 方針/ルールへの影響 | 各 BFF の責務範囲を policy として定義。BFF 間の共通ロジック共有方針（共有ライブラリ or 下流サービス呼び出し）を rule に含める |
| トレードオフ | フロントエンドごとの最適化・独立デプロイ ↔ コードの重複・BFF 数増加による運用コスト |
| 適用すべきでない場合 | フロントエンド種別間で API 要件がほぼ同一の場合。フロントエンドが1種類のみの場合 |

### Sidecar

| 項目 | 内容 |
|------|------|
| 概要 | アプリケーションの横断的関心事（ログ収集、メトリクス送信、認証プロキシ、サービスメッシュ等）を別プロセス/コンテナとしてデプロイする。メインアプリケーションとライフサイクルを共有するが、実装は独立する |
| RDRA シグナル | ティア数が多い場合（3ティア以上）で横断的関心事の統一が必要。マイクロサービス構成 |
| NFR シグナル | NFR C.1.3（監視範囲）、NFR C.6.1（ログ管理）。全サービスで統一的な可観測性が必要な場合 |
| ティアへの影響 | 各ティアのデプロイ構成（Pod 内にサイドカーコンテナを同梱） |
| 方針/ルールへの影響 | ティア共通方針として「サイドカーパターンによる横断的関心事の分離」を追加。サイドカーの責務範囲を rule に含める |
| トレードオフ | 横断的関心事の統一・言語非依存 ↔ リソースオーバーヘッド（各 Pod に追加コンテナ）・デバッグの複雑性 |
| 適用すべきでない場合 | モノリシックな単一アプリケーション。サービス数が少なくライブラリ共有で十分な場合 |

### Anti-Corruption Layer（ACL）

| 項目 | 内容 |
|------|------|
| 概要 | モダンアプリケーションとレガシーシステム（または異なるドメインモデルを持つ外部システム）の間にファサード/アダプタ層を設け、ドメインモデルの汚染を防ぐ。レガシーのデータ形式・プロトコルをモダン側のモデルに変換する |
| RDRA シグナル | 外部システム.tsv にレガシーシステムが存在する。外部システムのデータモデルが自システムと大きく異なる |
| NFR シグナル | NFR D（移行性）に関連。段階的なシステム移行時に特に重要 |
| ティアへの影響 | 外部連携ティアにアダプタ層追加。外部システムごとに ACL を設ける |
| 方針/ルールへの影響 | 外部連携ティアの policy として「ドメインモデル保護」を追加。ACL でのデータ変換ルールを rule に含める |
| トレードオフ | ドメインモデルの純粋性維持・レガシー依存の隔離 ↔ ACL 自体の開発・保守コスト |
| 適用すべきでない場合 | 外部システムのモデルが十分に整合的で変換が不要な場合 |

### Strangler Fig

| 項目 | 内容 |
|------|------|
| 概要 | レガシーシステムをマイクロサービスに段階的に移行する。レガシーの前段にファサード（プロキシ）を配置し、機能単位でモダンサービスに移行しながら、トラフィックを段階的に切り替える |
| RDRA シグナル | 外部システム.tsv にリプレース対象のレガシーシステムが存在する。BUC の一部がレガシーシステムで実現されている |
| NFR シグナル | NFR D（移行性）。無停止での段階的移行が要件の場合 |
| ティアへの影響 | API Gateway / リバースプロキシでのルーティング制御 |
| 方針/ルールへの影響 | 移行計画の policy として「段階的切り替え方針」を追加。機能単位の移行順序を rule に含める |
| トレードオフ | 低リスクの段階的移行・ビジネス継続性 ↔ 二重稼働期間のコスト・ルーティングの複雑性 |
| 適用すべきでない場合 | 小規模システムで完全リプレースのリスクが低い場合 |

### Ambassador

| 項目 | 内容 |
|------|------|
| 概要 | アプリケーションに代わってネットワークリクエストを処理するヘルパーサービス。リトライ、Circuit Breaker、TLS 終端等の通信関連の横断的関心事をアプリケーション外に委譲する |
| RDRA シグナル | 外部システム連携が多く、通信パターン（リトライ、タイムアウト等）の統一が必要 |
| NFR シグナル | NFR A.2.1（冗長化）、NFR C.1.3（監視） |
| ティアへの影響 | 外部連携ティアの通信層。Sidecar と組み合わせてデプロイ |
| 方針/ルールへの影響 | 外部連携ティアの policy として「通信パターンの一元管理」を追加 |
| トレードオフ | 通信ロジックの統一・アプリケーション簡素化 ↔ レイテンシの増加（プロキシ経由） |
| 適用すべきでない場合 | 通信パターンがシンプルで統一の必要がない場合 |

## 管理・監視パターン

### External Configuration Store

| 項目 | 内容 |
|------|------|
| 概要 | アプリケーション構成（接続文字列、機能フラグ、環境固有の設定等）を外部ストアで一元管理する。デプロイなしで構成変更を反映でき、環境間（開発/ステージング/本番）の構成管理を統一する |
| RDRA シグナル | ティア数が多い場合（3ティア以上）で構成の一元管理が有効。複数環境での運用が前提 |
| NFR シグナル | NFR C.2.1（計画保守）、NFR C.4.1（テスト環境） |
| ティアへの影響 | ティア共通ルールとして「構成外部化」を追加。構成ストアティアの追加を検討 |
| 方針/ルールへの影響 | 構成変更のバージョニング、ロールバック方針を rule に含める。シークレット管理（暗号化保存）を rule に含める |
| トレードオフ | 構成の一元管理・動的変更 ↔ 構成ストアの可用性要件・キャッシュとの整合性 |
| 適用すべきでない場合 | 単一サービスで環境変数管理が十分な場合 |

### Deployment Stamps

| 項目 | 内容 |
|------|------|
| 概要 | アプリケーションの独立したコピー（スタンプ）を複数リージョンまたは複数テナント向けにデプロイする。各スタンプは自己完結した独立ユニットとして動作し、スタンプ単位でスケール・更新・障害分離が可能 |
| RDRA シグナル | グローバル展開の要件がある場合。マルチテナントでテナントごとの完全なデータ分離が必要な場合 |
| NFR シグナル | NFR A.3.1（災害対策）>= Lv3。NFR A.2.1（冗長化）でリージョン分散が求められる場合 |
| ティアへの影響 | 全ティアのデプロイトポロジ。スタンプ間のトラフィックルーティング |
| 方針/ルールへの影響 | デプロイ方針として「スタンプ単位のデプロイ・更新」を追加。スタンプ間のデータ同期方針（必要な場合）を rule に含める |
| トレードオフ | リージョン冗長・テナント分離・独立スケール ↔ スタンプ数に比例する運用コスト・スタンプ間のデータ整合性 |
| 適用すべきでない場合 | 単一リージョンで十分な場合。テナント分離要件がない場合 |

## パターンの組み合わせ

推論時に以下の定番の組み合わせを考慮する。

### 外部連携の回復性スタック

| 組み合わせ | RDRA/NFR シグナル | 効果 |
|-----------|-----------------|------|
| Retry + Circuit Breaker + Timeout | 外部システム連携あり + NFR A.1.2（サービス切替時間） | 一時障害は再試行、継続障害は遮断、無限待ち防止。3つをセットで適用し、Timeout < Circuit Breaker の閾値時間とする |
| Retry + Circuit Breaker + Timeout + Bulkhead | 上記 + 外部システム3つ以上 | 上記に加え、外部システムごとにリソースプールを分離して障害影響を限定する |

### 分散トランザクションスタック

| 組み合わせ | RDRA/NFR シグナル | 効果 |
|-----------|-----------------|------|
| Saga + Compensating Transaction | 外部システムをまたぐ状態遷移 + NFR A.4.1（RPO/RTO） | 分散トランザクションの整合性確保 + 障害時の自動補正。Saga 適用時は必ず Compensating Transaction をセットで定義する |
| Saga + Compensating Transaction + Queue-Based Load Leveling | 上記 + 非同期処理容認 | Saga のステップ間をキューで接続し、負荷平準化と信頼性を向上する |

### 読み取り最適化スタック

| 組み合わせ | RDRA/NFR シグナル | 効果 |
|-----------|-----------------|------|
| CQRS + Event Sourcing | 状態遷移が多い + 監査ログ要件 + 読み書き非対称 | 変更履歴の完全保持 + 読み取り性能最適化。Event Sourcing のイベントから Materialized View を構築する |
| CQRS + Materialized View | 読み書き非対称 + 複雑な集約クエリ | Query 側に事前集計ビューを構築し、読み取り性能を最適化する |
| Cache-Aside + Static Content Hosting | 検索系 UC + 画像/ファイル属性あり | 動的データの読み取りキャッシュ + 静的コンテンツの CDN 配信で、全体的なレスポンスを改善する |

### 非同期処理スタック

| 組み合わせ | RDRA/NFR シグナル | 効果 |
|-----------|-----------------|------|
| Publisher-Subscriber + Queue-Based Load Leveling + Competing Consumers | バッチ/非同期処理あり + ピーク時対応 + NFR B.1.2 | 非同期イベント配信 + 負荷平準化 + 水平スケールによる並行処理 |
| Asynchronous Request-Reply + Queue-Based Load Leveling | 長時間処理あり + ユーザー体験の即時性が必要 | ユーザーには即時応答を返しつつ、バックグラウンドで負荷を平準化して処理する |
| Publisher-Subscriber + Claim Check | イベント通知 + 大容量データ連携 | イベント通知は MQ で軽量に行い、大容量データは Object Storage 経由で受け渡す |

### セキュリティスタック

| 組み合わせ | RDRA/NFR シグナル | 効果 |
|-----------|-----------------|------|
| Federated Identity + Gateway Offloading | 外部アクター + 認証/認可 + NFR E.5 | IdP への認証委譲 + Gateway でのトークン検証・認可の一元処理 |
| Valet Key + Static Content Hosting | ファイルアップロード/ダウンロード + 大容量データ | 署名付き URL による直接アクセスで Backend API の負荷を回避する |

## アンチパターン

推論時に以下のアンチパターンを回避する。

| パターン | アンチパターン | 問題 | 回避方法 |
|---------|-------------|------|---------|
| Retry | Retry Storm（全インスタンスが同時に再試行し、障害を悪化させる） | 障害中のサービスに大量のリトライが集中し、回復を妨げる | Jitter 付き指数バックオフを必須とする。最大リトライ回数を制限する |
| Circuit Breaker | 手動リセット未整備（Half-Open への遷移を自動のみに依存） | 障害復旧後もサーキットが開いたままになるリスク | 管理インターフェース（手動 Reset）を rule に含める |
| Circuit Breaker | 閾値の不適切な設定（敏感すぎる/鈍感すぎる） | 正常な一時的遅延で Open になる、または障害検知が遅れる | 外部システムの SLA に基づいた閾値設定を rule に含める |
| CQRS | 単純 CRUD への不要な適用（Over-Engineering） | 複雑性のみが増加し、恩恵がない | 読み書き非対称性を推論ルールで事前判定し、適用要否を確認する |
| Event Sourcing | 全エンティティへの無差別適用 | アーキテクチャ全体の複雑性が過度に増加する | 状態遷移の多さと監査要件の有無で適用対象を限定する |
| Saga | 補償トランザクション未実装（正常系のみ実装） | 障害時にデータ不整合が発生し、手動復旧が必要になる | Saga 適用時は補償処理を policy に必ず含める。補償不能なステップは最後に配置する |
| Bulkhead | モノリシック接続プール（全外部システムで共有） | 1つの外部システムの障害でプール枯渇→全体障害 | 外部システム数に応じてプール分離を推奨する |
| Cache-Aside | キャッシュ Stampede（キャッシュ失効時に全リクエストが DB に殺到） | DB の瞬間的な過負荷 | ミューテックス/ロック付きキャッシュ再構築、またはキャッシュの早期更新（stale-while-revalidate）を rule に含める |
| Queue-Based Load Leveling | DLQ 未監視（Dead Letter Queue に入ったメッセージを放置） | 処理失敗メッセージが消失し、データロスト | DLQ のモニタリングと手動再処理のオペレーション手順を rule に含める |
| Sharding | ホットスポット（特定シャードへのアクセス集中） | シャード間の負荷不均衡 | シャードキーの分散性を事前検証し、リバランシング方針を策定する |
