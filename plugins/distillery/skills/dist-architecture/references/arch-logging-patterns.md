# ロギングアーキテクチャ リファレンス

本ドキュメントは arch-inference-rules.md から参照され、ロギング方針の推論を支援する。

## ログカテゴリ

| カテゴリ | 説明 | 主な利用者 |
|---------|------|----------|
| アクセスログ | システムへの入口と出口の記録 | 運用担当者、SRE |
| 監査ログ | ビジネス上重要なイベントの不変な記録（誰が、いつ、何を、どうしたか） | セキュリティ担当、監査人 |
| 診断ログ | 内部フロー追跡、デバッグ用 | 開発者 |
| 依存関係ログ | 外部システムとの通信記録 | 開発者、SRE |

## レイヤー別ログ出力方針

### 基本原則

- 各レイヤーは自身の責務に対応するログカテゴリのみを出力する
- **domain 層は直接的なロギングを行わない**（ドメインイベントの発行または例外のスローで状態変化を通知する）
- ログの出力ポイントはレイヤーの境界（入口/出口）に集約する

### レイヤー × ログカテゴリ マトリクス

| レイヤー | アクセスログ | 監査ログ | 診断ログ | 依存関係ログ |
|---------|:----------:|:-------:|:-------:|:----------:|
| presentation | ◎ 主責務 | - | ○ 起動/終了 | - |
| usecase | - | ◎ 主責務 | ○ 機能開始/終了 | - |
| domain | - | - | - | - |
| gateway | - | △ 照合用 | ○ エラー/進捗 | ◎ 主責務 |

◎ = 主責務、○ = 副次的に出力、△ = 特定条件で出力、- = 出力しない

## エラーハンドリング伝播方針

例外の throw / catch / ログ出力の責務をレイヤーごとに定義し、多重ログを防止する。

### レイヤー x エラー責務マトリクス

| レイヤー | throw | catch+log | catch+rethrow | cause 保持 |
|---------|:-----:|:---------:|:-------------:|:----------:|
| presentation | - | - （HTTP ステータスに変換のみ） | - | - |
| usecase | - | **集約ポイント**（ここで 1 回だけログ出力） | - | Yes |
| domain | **発生源**（ドメイン例外をスロー） | - | - | - |
| repository | ラップ（技術例外→ドメイン例外） | - | Yes | Yes |
| gateway | 技術例外 | **依存関係ログ**（外部通信エラーのみ） | Yes | Yes |

### 原則

1. **集約ポイントで 1 回だけ**: エラーログは usecase 層（集約ポイント）で 1 回だけ出力する。中間レイヤー（repository）はキャッチ→ラップ→再スローのみで、ログを出力しない
2. **cause chain の保持**: 例外をラップする際は元例外を cause フィールドに保持し、集約ポイントのログ出力時に cause chain 全体を `context` に含める
3. **catch 握り潰し禁止**: catch 内では必ずログ出力または再スローを行う。何もしない空 catch は禁止
4. **gateway の依存関係ログ**: gateway 層は外部 DB/API の通信エラーを依存関係ログとして記録する。これは「集約ポイントのエラーログ」とは別カテゴリ（依存関係ログ）であり、多重ログには該当しない

### ティア別の適用

| ティア | 集約ポイント | HTTP/ジョブステータス変換 |
|--------|-----------|----------------------|
| Backend API | usecase 層 | presentation 層で HTTP ステータスに変換 |
| Worker (CronJob) | usecase 層 | presentation 層でジョブエラーログに変換 |
| Worker (MQ) | usecase 層 | presentation 層でメッセージ処理エラーに変換 |
| Frontend | 状態管理層 | ビュー層でユーザーフレンドリーなメッセージに変換 |

## アーキタイプ別ロギング戦略

### Web アプリケーション（Backend API ティア）

| カテゴリ | レベル | レイヤー | 内容 |
|---------|-------|---------|------|
| アクセスログ | ERROR | presentation | サーバーエラー（5xx）の HTTP メタデータ |
| アクセスログ | WARN | presentation | クライアントエラー（4xx）の HTTP メタデータ |
| アクセスログ | INFO | presentation | 成功した HTTP リクエスト/レスポンスのメタデータ（2xx, 3xx） |
| 監査ログ | INFO | usecase | 重要なビジネスイベントの記録（誰が、何を、どうしたか） |
| 診断ログ | INFO | usecase | 機能の開始・終了 |
| 依存関係ログ | INFO | gateway | 外部 DB/API 呼び出しの開始・終了、処理時間、成否 |
| 依存関係ログ | DEBUG | gateway | SQL クエリ、リクエスト/レスポンス本文 |
| (出力しない) | - | domain | ドメインイベントの発行または例外のスローで通知 |

### バッチジョブ（Worker ティア - CronJob）

| カテゴリ | レベル | レイヤー | 内容 |
|---------|-------|---------|------|
| 診断ログ | INFO | presentation | ジョブ起動、実行パラメータ、ジョブ終了（COMPLETED/FAILED） |
| 監査ログ | INFO | usecase | 重要なビジネスイベントの記録 |
| 診断ログ | INFO | usecase | 総実行時間、処理件数サマリー |
| 依存関係ログ | INFO | gateway | 外部 DB/API 呼び出しの処理時間、成否 |
| 診断ログ | ERROR | gateway | アイテム単位の処理エラー（対象アイテム、例外詳細） |
| 診断ログ | DEBUG | gateway | 定期的な処理進捗（ハートビート） |
| 監査ログ | INFO | gateway | 処理後の照合用集計情報（合計金額、チェックサム等） |
| (出力しない) | - | domain | ドメインイベントの発行または例外のスローで通知 |

### メッセージコンシューマ（Worker ティア - MQ）

| カテゴリ | レベル | レイヤー | 内容 |
|---------|-------|---------|------|
| 診断ログ | INFO | presentation | ストリーム処理の起動・停止 |
| 監査ログ | WARN | usecase | 重要なセキュリティイベントの検出 |
| 診断ログ | ERROR | usecase | 処理不能メッセージ（ポイズンピル）と例外情報 |
| 依存関係ログ | INFO | gateway | 集約されたメッセージ処理メトリクス |
| 依存関係ログ | WARN | gateway | パフォーマンス低下の兆候 |
| (出力しない) | - | domain | ドメインイベントの発行または例外のスローで通知 |

## WARN レベル劣化兆候ログ

正常動作だが劣化の兆候を示す状態を WARN レベルで記録する。放置すると障害に発展するリスクがある。

### 劣化兆候カテゴリ一覧

| カテゴリ | 兆候 | レイヤー | トリガー条件例 | デフォルトしきい値 |
|---------|------|---------|-------------|----------------|
| リトライ | リトライ開始 / 回数超過 | gateway | retry_count >= 1 | 1 回 |
| レートリミット | 残クォータ逼迫 | presentation | remaining_quota <= threshold | 残 10% |
| リソース | コネクションプール逼迫 | gateway | pool_usage >= threshold | 80% |
| サーキットブレーカー | 状態遷移（Closed→Open, Open→Half-Open） | gateway | state_change イベント | - |
| キャッシュ | キャッシュミス率上昇 | gateway | miss_rate >= threshold | 50% |
| キュー | キュー深度超過 | presentation (MQ) | queue_depth >= threshold | 要定義 |
| 認証 | トークン期限接近 | presentation | token_ttl <= threshold | 5 分 |
| 競合 | 楽観ロック競合（リトライ成功含む） | gateway | OptimisticLockException | - |
| 通信 | DNS/TLS ハンドシェイク遅延 | gateway | handshake_ms >= threshold | 要定義 |

### しきい値の外部設定化

トリガー条件のしきい値（remaining_quota, pool_usage, miss_rate, queue_depth, token_ttl, handshake_ms 等）はコードにハードコードせず、**設定ファイルまたは環境変数**から読み込む。上表の「デフォルトしきい値」は初期値として設定ファイルに定義し、運用環境に合わせて調整可能にする。

### WARN ログの構造化フォーマット

劣化兆候ログは通常の構造化ログ必須フィールドに加え、以下を `context` に含める:

| フィールド | 説明 | 例 |
|-----------|------|-----|
| degradation_type | 劣化カテゴリ名 | `"retry"`, `"circuit_breaker"`, `"pool_exhaustion"` |
| current_value | 現在の計測値 | `3` (リトライ回数), `85` (プール使用率%) |
| threshold | しきい値 | `80` (プール使用率閾値%) |
| action_taken | 実行したアクション | `"retry_with_backoff"`, `"circuit_opened"` |

## 構造化ログの必須フィールド

| フィールド | 説明 |
|-----------|------|
| timestamp | ISO 8601 形式（UTC 統一） |
| level | INFO, WARN, ERROR, DEBUG, TRACE |
| trace_id | リクエスト横断の追跡 ID（OpenTelemetry 標準） |
| span_id | ティア単位の処理 ID（OpenTelemetry 標準） |
| service | サービス名 + バージョン |
| message | 静的テキスト（動的値は context に分離） |
| context | 追加情報のキーバリューペア（下記参照） |

## トレーサビリティ ID 体系

OpenTelemetry 標準の分散トレーシング ID に統一する。業務固有の属性はログの `context` フィールドに含める。

### 分散トレーシング ID（OpenTelemetry 標準）

| フィールド | 説明 | 生成ポイント | 伝播方式 |
|-----------|------|-----------|---------|
| trace_id | リクエスト横断の追跡 ID。1 リクエストに 1 つ | フロントエンド / API Gateway | HTTP ヘッダー（W3C Trace Context）/ MQ メタデータ |
| span_id | ティア単位の処理 ID。ティア間通信で新 span を開始し、parent-child 関係で因果を表現 | ティアの入口（API Gateway, Backend API, Worker 等） | OpenTelemetry SDK |

### 粒度の原則

span はティア単位で発行する。レイヤー内の処理詳細は span を切らず、ログの `context` フィールドで表現する。

### ログ context に含める業務属性

| フィールド | 説明 | 設定ポイント |
|-----------|------|-----------|
| session_id | セッション ID | presentation（認証セッションから取得） |
| user_id | 操作ユーザー ID | presentation（認証後にセット） |
| 業務エンティティ ID | 予約 ID、精算 ID 等の業務単位 ID | usecase（業務処理開始時にセット） |

イベント駆動の因果チェーンは span の parent-child 関係で表現する。業務単位の横断検索は `context` 内の業務エンティティ ID で行う。

## セキュリティ制約

ログに記録してはいけない情報:
- 認証情報（パスワード、API キー、アクセストークン）
- PII（マイナンバー、健康情報、メールアドレス、電話番号）
- 金融データ（クレジットカード番号、銀行口座情報）

必要時はマスキング/ハッシュ化を適用する。

## ログ運用設計

| 運用項目 | 方針 | NFR シグナルとの対応 |
|---------|------|------------------|
| ログローテーション | サイズベース（100MB）+ 時間ベース（日次）の併用。しきい値は設定ファイルで調整可能 | NFR C.6.1 ログ保管期間 |
| 保持期間 | アクセスログ / 診断ログ: NFR C.6.1 の Lv に準拠。監査ログ: NFR E.7.1 に準拠（通常は最長） | NFR C.6.1, NFR E.7.1 |
| 動的ログレベル変更 | 再起動なしでログレベルを変更可能な仕組み（設定ファイル or API） | NFR C.3.1 障害検知 >= Lv3 |
| パフォーマンス影響 | 非同期ログ出力を原則とする。DEBUG/TRACE は本番環境で無効がデフォルト | NFR B.2.1 レスポンスタイム |
| ログ出力先 | stdout/stderr に統一（ベンダーニュートラル）。ログ収集・集約は基盤レイヤーの責務 | - |

## ログアンチパターン

実装時に回避すべきアンチパターンと対策。

| アンチパターン | 説明 | 対策 | 関連ルール |
|-------------|------|------|-----------|
| 多重ログ | 同一例外を複数レイヤーで重複ログ出力 | エラーハンドリング伝播方針に従い集約ポイントで 1 回のみ | エラーハンドリング伝播方針 |
| catch 握り潰し | 例外をキャッチしてログもスローもしない | catch 時は必ずログ出力 or 再スロー。空 catch を禁止 | エラーハンドリング伝播方針 |
| 機密情報漏洩 | PII・認証情報をログに平文出力 | セキュリティ制約のマスキングルールを適用 | セキュリティ制約 |
| 大量ログ | ループ内での逐次ログ、DEBUG レベルの本番有効化 | ループはサマリログ（N 件中 M 件失敗等）。本番は INFO 以上がデフォルト | ログ運用設計 |
| 非構造化ログ | 文字列連結による自由形式ログ | JSON 構造化ログの必須フィールドを強制 | 構造化ログの必須フィールド |
| TZ 不統一 | サーバーごとに異なるタイムゾーンでタイムスタンプ出力 | 全ログを UTC（ISO 8601）で統一。表示時にローカル TZ 変換 | 構造化ログの必須フィールド |

## RDRA/NFR シグナルとの対応

| シグナル | ロギング方針への影響 |
|---------|------------------|
| NFR C.6.1 ログ保管期間 | 保管期間の設定。監査ログは長期保管 |
| NFR C.1.3 監視範囲 | アクセスログ + 依存関係ログの出力レベル |
| NFR E.7.1 監査ログ | 監査ログの出力範囲と保管ポリシー |
| NFR E.6.1 データ暗号化 | PII のマスキング/ハッシュ化ルール |
| 外部システム.tsv | 依存関係ログの出力対象 |
| 状態.tsv の状態遷移 | 監査ログの出力対象イベント |
| NFR C.3.1 障害検知方式 | WARN レベル劣化兆候ログの出力範囲。Lv3 以上で動的レベル変更を推奨 |
| NFR B.2.1 レスポンスタイム | ログ出力のパフォーマンス影響考慮。非同期ログ出力を推奨 |
| BUC.tsv のバッチ/非同期処理 | 劣化兆候ログの対象（キュー深度、リトライ等） |
