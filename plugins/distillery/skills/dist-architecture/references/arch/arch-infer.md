# Step1: RDRA + NFR からのアーキテクチャ推論

RDRA モデルと NFR グレードを読み取り、アーキテクチャ設計を推論するタスク。

## 入力

- `docs/rdra/latest/*.tsv` — 現在の RDRA モデル（全ファイル）
- `docs/rdra/latest/システム概要.json` — システム概要
- `docs/nfr/latest/nfr-grade.yaml` — NFR グレード
- `references/arch-inference-rules.md` — 推論ルール
- `references/arch-schema.md` — 出力スキーマ定義
- `docs/arch/latest/arch-design.yaml`（差分更新モード時のみ）— 既存のアーキテクチャ設計

## タスク手順

### 1. RDRA モデルの読み込みと特徴抽出

以下の観点で RDRA モデルを分析する:

#### BUC.tsv から
- 業務数と UC 数（システムの規模感）
- バッチ/タイマー処理の有無（ワーカーティアの要否）
- 通知・メール送信アクティビティの有無（非同期処理の要否）
- 業務間の依存関係（API 粒度の判断材料）
- 「多言語」「海外」「グローバル」等のキーワードの有無（i18n 方針の判断材料）

#### アクター.tsv から
- アクター種別数と社内外区分（フロントエンドティアの要否）
- ロールの多様性（認証/認可方式の判断材料）
- 外部アクターの有無（UI の複雑さ）
- 外国語名のアクターの有無（i18n 要否の判断材料）

#### 外部システム.tsv から
- 外部システム数と種類（連携ティアの要否）
- 決済・認証等の重要な連携（冪等性・セキュリティ要件）
- 海外サービスとの連携有無（i18n 要否の判断材料）

#### 情報.tsv から
- エンティティ数と属性数（データモデルの規模）
- 個人情報・機密情報の有無（セキュリティアーキテクチャ）
- 金銭・取引関連の情報（トランザクション要件）
- エンティティ間の関連（リレーション構造）

#### 状態.tsv から
- 状態遷移の複雑さ（ドメインモデルの要否）
- 状態数と遷移パス数（レイヤリング判断材料）

#### 条件.tsv から
- ビジネスルールの数（ドメイン層の複雑さ）
- アクセス制御・権限条件（認可アーキテクチャ）
- バリデーション条件（入力検証方針）

#### バリエーション.tsv から
- バリエーション数と選択肢の多様性（ストラテジーパターンの検討）
- 言語・地域バリエーションの有無（i18n 要否の判断材料）

#### システム概要.json から
- システムの目的・概要
- 対象ユーザー層
- 海外・グローバル展開の言及有無（i18n 方針の判断材料）

### 2. NFR グレードの読み込みと影響分析

nfr-grade.yaml からカテゴリ別に主要メトリクスを抽出し、アーキテクチャへの影響を分析する:

| NFR カテゴリ | 分析観点 |
|-------------|---------|
| A（可用性） | 冗長構成、デプロイトポロジ、ヘルスチェック |
| B（性能・拡張性） | キャッシュ戦略、非同期処理、スケーリング方式 |
| C（運用・保守性） | 監視方式、ログ方針、CI/CD、バックアップ |
| D（セキュリティ） | 認証/認可、暗号化、監査ログ |
| E（移行性） | デプロイ戦略（Blue/Green, Canary 等） |
| F（環境） | モバイル対応、ブラウザ対応 |

### 3. システムアーキテクチャの推論

`arch-inference-rules.md` の「Part 1: システムアーキテクチャ推論ルール」に従い:

1. **ティア識別**: 各ティア（フロントエンド、バックエンド API、ワーカー、データストア、外部連携）の要否を判定
2. **テクノロジー候補**: 各ティアのベンダーニュートラルなテクノロジー候補を推論
3. **ティアごとの方針・ルール**: RDRA/NFR から導出される方針・ルールを設定
4. **ティア共通方針・ルール**: ティアを横断する方針・ルールを設定
5. **Mermaid 図**: ティア構成図を `graph TD` 形式で生成
6. **i18n 方針**: RDRA シグナルから多言語対応の要否とアーキテクチャ方針を推論

### 4. アプリケーションアーキテクチャの推論

`arch-inference-rules.md` の「Part 2: アプリケーションアーキテクチャ推��ルール」に従い:

1. **レイヤリングパターン選定**: 各ティアのレイヤリング（2層/3層/4層）を判定
2. **レイヤー定義**: 各レイヤーの責務と依存関係を設定
3. **レイヤーごとの方針・ルール**: RDRA/NFR から導出される方針・ルールを設定
4. **レイヤー共通方針・ルール**: レイヤーを横断する方針・ルールを設定（ティア内）
5. **Mermaid 図**: ティアごとのレイヤー依存図を `graph TD` 形式で生成

### 5. データアーキテクチャの推論

`arch-inference-rules.md` の「Part 3: データアーキテクチャ推論ルール」に従���:

1. **エンティティ抽出**: 情報.tsv の各行からエンティティを生成
2. **属性の型推論**: 属性名パターンから論理型を推論
3. **リレーション推論**: 関連情報からカーディナリティを推論
4. **ストレージマッピング**: 各エンティティのストレージ種別を推論
5. **Mermaid 図**: 概念モデルを `erDiagram` 形式で生成

### 6. confidence の付与

全ての方針・ルール・マッピングに confidence を付与する:
- `high`: RDRA/NFR から直接推論できた
- `medium`: RDRA/NFR から間接的に推論した
- `low`: 弱い根拠での推論
- `default`: 一般的なベストプラクティスを適用
- confidence は保守的に付与し、曖昧な場合は低めにする

### 8. source_model への NFR ID 明記

全ての policy/rule の `source_model` フィールドに、根拠となった NFR メトリクス ID を明記する:

- 形式: `"NFR {ID}"` （例: `"NFR A.1.1.1"`, `"NFR E.5.2.1"`）
- 複数の NFR が根拠の場合はカンマ区切り（例: `"NFR A.2.1.1, NFR A.2.5.1"`）
- RDRA 要素も根拠の場合は併記（例: `"アクター: 利用者, NFR E.5.1.1"`）
- 全ての重要メトリクス（important: true）が少なくとも1つの policy/rule の source_model で参照されていることを目標とする
- カバレッジレポートで RDRA 網羅率・NFR 網羅率ともに 100% を達成すること

### 7. 差分更新モード時の処理

既存の `docs/arch/latest/arch-design.yaml` がある場合:

1. RDRA の差分（`docs/rdra/events/{最新event_id}/_changes.md`）または NFR の差分を読み取る
2. 変更された要素に関連するアーキテクチャ項目を特定する
3. 該当項目のみを再推論する
4. 既存設計の `confidence: "user"` の項目は再推論しない（ユーザー確定値を尊重）
5. 再推論結果と既存値の差異をリストアップする

## 出力

このステップではファイル出力を行わない。推論結果は Step2（対話）に渡すための内部データとして保持する。

## 注意事項

- 推論は保守的に行う（迷ったらシンプルな構成を推定し、Step2 で複雑化の判断を委ねる）
- テクノロジー候補はベンダーニュートラルな用語のみ使用（arch-schema.md のベンダーニュートラル用語ガイド参照）
- RDRA/NFR に記載のない情報を想像で補完しない
- 推論根拠は具体的な RDRA/NFR 要素名を引用する
- Mermaid 図はシンプルで読みやすい構成にする
