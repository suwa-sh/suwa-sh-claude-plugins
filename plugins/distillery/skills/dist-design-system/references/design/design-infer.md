# Step1: モデル分析とデザイン方針決定

## タスク概要

RDRA/NFR/Arch/Infra モデルを読み込み、デザインシステムの方針を決定する。

## 入力ファイル読み込み

### 1. RDRA モデル分析

**BUC.tsv から画面・ポータル構成を抽出:**

```
BUC.tsv の構造:
業務 | BUC | 先 | アクティビティ | 次 | UC | 関連モデル1 | 関連オブジェクト1 | 関連モデル2 | 関連オブジェクト2 | 説明 | 立場
```

- `関連モデル1 = "画面"` の行から画面名を抽出 → ユニーク画面一覧
- `関連モデル2 = "アクター"` の行からアクター名を抽出 → 画面-アクター紐付け

**アクター.tsv からポータル導出:**

```
アクター.tsv の構造:
アクター群 | アクター | 役割 | 社内外 | 立場 | 主担当業務
```

- 社内外 + 立場 の組み合わせでポータルを導出:
  - 社外 + 受益者 → User ポータル (Blue)
  - 社外 + 提供者 → Provider ポータル (Teal)
  - 社内 + 提供者 → Admin ポータル (Slate)
- ポータルが2つの場合は Blue + Slate を推奨

**情報.tsv からデータ表示コンポーネントを導出:**

```
情報.tsv の構造:
コンテキスト | 情報 | 属性 | 関連情報 | 状態モデル | バリエーション | 説明
```

- 属性にカンマ区切りのフィールド一覧 → Card/Detail コンポーネントの Props
- 状態モデルが存在する情報 → StatusBadge/Tracker コンポーネント候補
- 金額関連属性 → PriceDisplay コンポーネント
- 評価関連属性 → StarRating コンポーネント
- 画像関連属性 → ImageCard コンポーネント

**状態.tsv からステータス表示を導出:**

```
状態.tsv の構造:
コンテキスト | 状態モデル名 | 状態 | 次の状態 | イベント | 説明
```

- 各状態モデルの状態一覧 → Badge variant 定義
- 状態遷移 → Tracker/Stepper コンポーネント候補
- 状態数が 3 以上 → ステップトラッカーを検討

**バリエーション.tsv からフィルター/フォームを導出:**

- バリエーション名が「〜条件」→ SearchFilter のフィルターグループ
- バリエーション名が「〜種別」→ Select/Radio コンポーネント
- バリエーション名が「〜区分」→ Tab/SegmentedControl コンポーネント

### 2. NFR → デザイン判断

| NFR 項目 | デザイン判断 |
|----------|-------------|
| 可用性 (24h運用等) | エラー状態・ローディング状態を全画面に |
| 性能 (同時ユーザー数) | ページネーション・仮想スクロール要否判定 |
| 性能 (レスポンスタイム) | Skeleton UI でパーシーブドパフォーマンス改善 |
| セキュリティ (RBAC) | ポータル別ナビゲーション・権限ベース表示切替 |
| セキュリティ (PII) | マスク表示、オートコンプリート制御 |
| 運用 (モバイル対応) | User ポータルをモバイルファースト設計 |

### 3. Arch → 技術判断

| Arch 要素 | デザイン判断 |
|-----------|-------------|
| presentation tier の言語/FW | コンポーネント実装技術 (React/Vue等) |
| ポータル数 | テーマ切替方式 (data-portal 属性) |
| IdP (Cognito等) | ログイン画面は Hosted UI or カスタム |
| CDN | Static asset キャッシュ戦略 |
| API Gateway Rate Limit | スロットリング時のリトライ UI |
| Cache TTL | stale-while-revalidate パターン |

### 5. レイアウト・スペーシング推論

画面構成や情報密度から、グリッドレイアウトとスペーシングの具体値を根拠付きで導出する。
推論結果は `_inference.md` に根拠とともに記録し、Step3 のトークン生成で参照する。

#### 5-1. グリッドシステム推論

**基本構造:** 12カラムグリッドを基本とする（Tailwind CSS の標準グリッドと整合）。

**サイドバー幅の推論:**

サイドバー幅はナビゲーション項目数（= 業務数 + 共通メニュー）から導出する。

| 推論入力 | 導出元 | 推論ルール |
|---------|--------|-----------|
| ナビ項目数 | BUC.tsv の業務ユニーク数 + 共通メニュー（ホーム, 設定等） | 項目数 ≤ 8: コンパクト (12rem)。項目数 9〜15: 標準 (16rem)。項目数 > 15: ワイド (20rem) |
| ラベル文字数 | 業務名の最大文字数 | 全角8文字以上: 標準以上を推奨（文字切れ防止） |
| collapsed 幅 | - | アイコン + 余白で固定 (4rem) |

**根拠の記録例:**
```
サイドバー幅: 16rem
  根拠: 業務数 6 + 共通メニュー 2 = ナビ項目 8。標準幅で十分。
  最大ラベル「利用者管理業務」= 全角7文字、16rem で収まる。
collapsed 幅: 4rem
  根拠: アイコン 24px + padding 左右 20px = 64px = 4rem（固定値）
```

**画面パターン別カラム分割の推論:**

各画面のカラム分割は、画面の用途と表示する情報量から導出する。

| 画面用途 | 推論入力 | カラム分割 | 推論ルール |
|---------|---------|-----------|-----------|
| 一覧・検索 | 情報.tsv の属性数 | 12col (フル幅) | 属性数 ≥ 5 でテーブル表示が必要 → フル幅 |
| 詳細表示 | 関連情報の有無 | 8col + 4col (マスター・ディテール) | 関連情報あり → サイドパネルに関連データ。なし → フル幅 |
| 登録・編集フォーム | 入力フィールド数 | 余白2col + 8col + 余白2col (中央寄せ) | フィールド数 ≤ 10 → 中央寄せ。> 10 → セクション分割でフル幅 |
| ダッシュボード・統計 | 集計項目数 | 6col + 6col (2段) or 4col × 3 | KPI カード数から格子パターンを決定 |
| ワーカー系（画面なし） | - | - | 画面なし |

**根拠の記録例:**
```
蔵書検索画面: 12col (フル幅)
  根拠: 書籍の属性数 8（タイトル,著者,ISBN,...）、テーブル表示が最適。
利用者登録画面: 余白2col + 8col + 余白2col
  根拠: 入力フィールド 5（氏名,連絡先,メール,種別,メモ）、中央寄せフォームで十分。
統計レポート画面: 4col × 3 (上段KPI) + 12col (下段チャート)
  根拠: KPIカード 3件（貸出数,延滞数,予約数）+ 時系列チャート 1件。
```

**レスポンシブ推論:**

ブレークポイントは Arch 設計の NFR F（環境適合性）から導出する。

| NFR 入力 | 推論ルール |
|---------|-----------|
| F.1.1.1 端末 Lv1 (PC のみ) | lg のみ設計。md/sm は簡易対応 |
| F.1.1.1 端末 Lv2 (PC + タブレット) | lg + md を設計。sm は簡易対応 |
| F.1.1.1 端末 Lv3 (PC + タブレット + スマホ) | lg + md + sm をフル設計（モバイルファースト） |

| ブレークポイント | カラム数 | サイドバー | 推論根拠 |
|----------------|---------|-----------|---------|
| lg (1024px+) | 12col | 展開 | デスクトップ標準 |
| md (768-1023px) | 8col | 折りたたみ (collapsed 幅) | タブレット横持ち相当 |
| sm (<768px) | 4col | 非表示 (ハンバーガーメニュー) | スマホ相当 |

#### 5-2. スペーシング推論

スペーシングの具体値は 4px ベースグリッド（Primitive トークン）から選択し、用途ごとにセマンティックトークンとして定義する。

**セマンティック spacing トークンの推論:**

| トークン名 | 用途 | 推論ルール | 推論入力 |
|-----------|------|-----------|---------|
| page-padding | ページ全体の内側余白 | サイドバーありの管理画面: 1.5rem (6)。サイドバーなしの公開画面: 1rem〜2rem (4〜8) | ポータル構成 |
| section-gap | セクション間の縦方向間隔 | 情報密度が高い（属性数平均 ≥ 6）: 2rem (8)。低い: 3rem (12) | 情報.tsv の属性数平均 |
| component-gap | コンポーネント内の要素間隔 | 標準: 0.75rem (3)。密: 0.5rem (2) | 画面あたりの表示要素数 |
| card-padding | カードの内側余白 | 属性数 ≤ 4: 1rem (4)。属性数 ≥ 5: 1.5rem (6) | 情報.tsv のカード候補の属性数 |

**根拠の記録例:**
```
page-padding: 1.5rem (spacing-6)
  根拠: 司書ポータル（サイドバーあり）のメインコンテンツ領域。
  情報密度が高い管理画面のため、余白を控えめにして表示領域を確保。
section-gap: 2rem (spacing-8)
  根拠: 情報.tsv の属性数平均 6.2（書籍8, 利用者5, 貸出5, 予約4, 統計8）。
  情報密度が高いため、セクション間は 2rem で視覚的分離を確保。
component-gap: 0.75rem (spacing-3)
  根拠: フォーム要素やリスト項目の標準間隔。4px グリッドの 3倍。
```

#### 5-3. スペーシング適用方針

以下の方針を `_inference.md` に記録し、Storybook 実装で一貫して適用する。

**原則: gap > padding > margin**

| 手段 | 使う場面 | 理由 |
|------|---------|------|
| `gap` (flex/grid) | 兄弟要素間の間隔 | 親コンテナで一括制御。子要素に余計なスタイルが漏れない |
| `padding` | コンテナの内側余白 | 要素自身の領域を定義。背景色・ボーダーとの関係が明確 |
| `margin` | **原則不使用** | コンポーネント間の結合を生む。使う場合はセクション間（`section-gap` トークン）に限定 |

**実装パターン:**

```
/* 良い例: gap で子要素間隔を制御 */
.card-list { display: grid; gap: var(--component-gap); }

/* 良い例: padding でコンテナ内側余白 */
.card { padding: var(--card-padding); }

/* 良い例: セクション間のみ margin 許容 */
.section + .section { margin-top: var(--section-gap); }

/* 悪い例: 子要素に margin を付けて間隔制御 */
.card { margin-bottom: 1rem; }  /* → gap に置き換える */
```
