# アプリケーションアーキテクチャ リファレンス

本ドキュメントは arch-inference-rules.md の Part 2 から参照され、ティアごとのレイヤリング選定を支援する。

## 設計思想

アプリケーションアーキテクチャの目的は一貫している:

1. **ドメインロジックと入出力を分離する** — ドメインに集中できるようにする
2. **入出力を相手ごとに分離し定型化する** — Driver Side（利用する側）と Driven Side（利用される側）
3. **変更があったときに凹型で依存を内側に向ける** — IF 導入は必要になってから

## 基本レイヤー構成（4層）

| レイヤー | 責務 | Driver/Driven | 依存先 |
|---------|------|:------------:|--------|
| presentation | Driver Side の入出力。HTTP リクエスト/レスポンス変換、入力バリデーション | Driver | usecase |
| usecase | ビジネスフロー制御。ドメインロジックとデータ入出力の呼び出しで完結する | - | domain, gateway |
| domain | ドメインロジック（計算・判定ロジック）。エンティティ、値オブジェクト、ドメインイベント | - | なし |
| gateway | Driven Side の入出力。データストアアクセス、外部システム連携、MQ 発行 | Driven | domain |

### IF（インターフェース）の導入方針

**初期状態: IF なし（直接依存）**

```
presentation → usecase → domain
                       → gateway → domain
```

- usecase は gateway を直接利用する
- 開発スピードを優先し、凹型は導入しない

**凹型への移行（必要になった場合）**

```
presentation → usecase → domain
                       → gateway(IF) ← gateway(実装) → domain
```

以下の場合に凹型（IF 導入）を検討する:
- 外部サービス API の変更が頻繁に発生する
- データストア製品の乗り換えが予定されている
- 開発チームをレイヤーで分割する必要がある
- テスト時に mock を利用したい gateway がある

### IF なしの前提条件

以下が成り立つ場合、IF なしで開始する合理性がある:
- usecase の API は頻繁に増減するが、変更はそうそうしない
- 利用する外部サービスやデータストア製品は、そうそう乗り換えない
- 外部サービスの API 変更は稀にあるが、それはがんばりどころ
- データモデルは変更が起きにくく設計しておく

## システムアーキテクチャごとのバリエーション

### webapp backend（Backend API ティア）

| レイヤー | 具体的な責務 |
|---------|------------|
| presentation | HTTP リクエストのパース、レスポンス変換。Web FW のハンドラ |
| usecase | ビジネスフロー制御、トランザクション境界 |
| domain | ビジネスルール、エンティティ、状態遷移 |
| gateway | ORM によるデータアクセス、外部 API クライアント（OpenAPI 生成コード等） |

### PubSub subscriber / MQ consumer（Worker ティア）

| レイヤー | 具体的な責務 |
|---------|------------|
| presentation | メッセージのデシリアライズ、サブスクライバハンドラ |
| usecase | 非同期処理のビジネスフロー制御 |
| domain | ビジネスルール、状態遷移（Backend API と共有可能） |
| gateway | データアクセス、外部 API クライアント（Backend API と共有可能） |

### CLI（バッチ・管理ツール）

| レイヤー | 具体的な責務 |
|---------|------------|
| presentation | CLI ライブラリによる引数パース、出力フォーマット |
| usecase | バッチ処理のフロー制御 |
| domain | ビジネスルール |
| gateway | ファイルアクセス、SDK 直接利用（簡易な場合は gateway 省略可） |

### フロントエンド SPA

| レイヤー | 具体的な責務 |
|---------|------------|
| presentation (view/component) | UI 描画、ユーザー操作のハンドリング |
| usecase (state management) | アプリケーション状態管理、画面間の状態共有 |
| domain | （フロントエンドでは薄い。バリデーション等） |
| gateway (api client) | Backend API との通信、認証トークン管理 |

## 参考にしているアーキテクチャパターン

各パターンは「やりたいこと」が一貫している。本スキルでは4層構成を基本とし、パターン名は参考情報として記載する。

| パターン | 特徴 | 本スキルとの対応 |
|---------|------|---------------|
| レイヤードアーキテクチャ | 上から下への一方向依存 | 基本構成 |
| 凹型レイヤー | IF と実装のレイヤー分離で依存逆転 | IF 導入時の構成 |
| ヘキサゴナルアーキテクチャ | Port & Adapter。Driver Side / Driven Side の分類 | presentation(Driver) / gateway(Driven) の対応 |
| オニオンアーキテクチャ | Application Core（usecase + domain）が内側 | usecase + domain の分離 |
| クリーンアーキテクチャ | 同心円で外側から内側への依存。必要な時に凹型で逆転 | 成長戦略の考え方 |
