# Step2: 対話によるアーキテクチャ設計確認・調整

Step1 の推論結果をユーザーに提示し、選択肢形式の対話で確認・調整するタスク。

## 入力

- Step1 の推論結果サマリ（内部データ）
- `references/arch-schema.md` — スキーマ定義の参照用

## 対話の基本方針

1. **選択肢形式**: すべての確認は番号付き選択肢で提示し、ユーザーは番号で回答できるようにする
2. **段階的な確認**: テクノロジースタック → システムアーキテクチャ → アプリケーションアーキテクチャ → データアーキテクチャ の順で進める
3. **Mermaid 図の活用**: 各 Phase で推論した Mermaid 図を提示し、視覚的に理解を助ける
4. **変更の記録**: ユーザーが変更した項目は confidence: "user" に更新する
5. **ベンダーニュートラル**: テクノロジー候補は FaaS, CaaS(k8s), RDB 等のベンダーニュートラルな用語で提示する

## 対話フロー

### Phase 0: テクノロジースタックの確認

言語/フレームワークの希望と技術的制約をヒアリングする。デプロイ先（クラウドベンダー）は本スキルの対象外のため確認しない。

```
## テクノロジースタックの確認

RDRA モデルと NFR グレードからアーキテクチャを推論するにあたり、技術スタックの希望を確認します。

### Q1. 使用言語の希望はありますか？

1. **TypeScript**（フロントエンド・バックエンド統一）
2. **Go**（バックエンド）+ **TypeScript**（フロントエンド）
3. **Java/Kotlin**（バックエンド）+ **TypeScript**（フロントエンド）
4. **Python**（バックエンド）+ **TypeScript**（フロントエンド）
5. **その他**（具体的に記述）
6. **未定・おまかせ**

### Q2. 技術的制約はありますか？

1. **特になし**
2. **制約あり**（具体的に記述: 例「既存DBとの互換性」「モノレポ構成」）

**→ 各質問の番号を回答してください（例: 「1, 1」）**
```

ユーザーの回答を technology_context に反映する。

> 💡 この選択は決定記録（arch-decision-{NNN}.yaml）として記録される

## 出力

ファイル出力はこのステップでは行わない。

## 対話時の注意事項

- すべての確認は **番号選択式** で行う。自由記述を求めるのは変更内容の指定のみ
- ユーザーが Phase 0 で「6, 1」（未定・制約なし）等と即答した場合は、推論結果をそのまま使用
- テクノロジー候補はベンダーニュートラルな用語のみ（AWS, Azure, GCP 等の具体名は使用しない）
- 対話は最大3ラウンドを目安とし、収束しない場合は現状で確定を提案する
- 差分更新モードでは、変更があった項目のみを提示する
- Mermaid 図はコードブロック（```mermaid）で囲んで提示��る
