# データアーキテクチャ リファレンス

本ドキュメントは arch-inference-rules.md の Part 3 から参照され、データモデリング方針の選定を支援する。

## データモデリング方針

### ミュータブル vs イミュータブル

| 方針 | 概要 | 適合ケース |
|------|------|----------|
| ミュータブル（従来型） | UPDATE/DELETE を許容。nullable カラムで状態変更を表現 | CRUD 中心、状態遷移が少ない、監査要件なし |
| イミュータブル（スナップショット併用型） | イベントは INSERT のみ、スナップショットで最新状態を参照 | 状態遷移が多い、金銭取引、監査要件あり |

### 選定基準

| 判定条件 | 推奨方針 | confidence |
|---------|---------|-----------|
| 状態.tsv に状態モデルがあるエンティティ | event_snapshot | high |
| 個人情報（氏名・連絡先等）があり変更履歴の追跡が一般的 | event_snapshot | medium |
| 「〜する」動詞で表現でき、発生日時を持ち、登録後に変更しない | event | high |
| 金銭関連（金額・料金属性）で一度きりの記録 | event | high |
| 属性変更が想定され世代管理が必要なマスタ | resource_scd2 | medium |
| 単純なマスタ・設定、上記いずれにも該当しない | resource_mutable | default |

### テーブル分類

RDRA の情報.tsv の各エンティティを以下の4種類に分類する:

| 分類 | 特徴 | RDRA シグナル | テーブル設計 |
|------|------|-------------|-----------|
| Event + Snapshot (event_snapshot) | 状態遷移するエンティティ。イベントで遷移を記録、スナップショットで最新状態をキャッシュ | 状態.tsv に状態モデルがある、または属性変更の履歴追跡が必要 | {entity}_events (INSERT のみ) + {entity}_snapshots (最新状態) |
| Event (event) | 一度きりの事実の記録。登録後は変更しない | 「〜する」動詞で表現でき、発生日時を1つ持ち、登録後に変更しない | {entity} テーブル (INSERT のみ)。スナップショット不要 |
| Resource SCD2 (resource_scd2) | 世代管理が必要なマスタ | 属性変更が想定され適用期間での管理が必要 | valid_from / valid_to |
| Resource Mutable (resource_mutable) | 単純なマスタ・設定データ | 変更頻度が低い | 従来型テーブル |

### モデリングの5ステップ（RDRA 対応）

| ステップ | 内容 | RDRA モデルからの推論 |
|---------|------|-------------------|
| Step1: エンティティ抽出 | 業務名詞の洗い出し | 情報.tsv の各行がエンティティ |
| Step2: リソースとイベントの分類 | 動詞→イベント、名詞→リソース | 状態.tsv に状態モデルがある → イベント型、ない → リソース型 |
| Step3: イベントの正規化 | 1イベント = 1日時属性 | 状態.tsv の各遷移が1イベント型に対応 |
| Step4: リソースのイミュータブル化 | nullable 日時 → 隠れたイベント | 情報.tsv の属性に「〜日」「〜日時」が複数 → イベント分離を検討 |
| Step5: 関係性の履歴管理 | 変化する関係 → 交差エンティティ | 情報.tsv の関連情報で N:M や変化する関係 → 交差エンティティ |

### スナップショット併用型の設計方針

| 論点 | 推奨方針 | 理由 |
|------|---------|------|
| 最新状態の取得 | スナップショット併用 | RDB の検索性能を活かしつつイベント履歴を保持 |
| 訂正・削除 | 訂正・論理削除イベント | 物理削除を行わず、完全な監査証跡を維持 |
| マスタ世代管理 | 適用期間モデル（valid_from/valid_to） | SQL の JOIN による期間結合が可能 |
| 長期プロセスの状態 | キャッシュ的ステータス | 親エンティティに current_status を持たせ業務クエリを高速化 |
| 永続化の実装 | gateway 層で二重書き込みを隠蔽 | usecase は「保存」のみ、gateway が INSERT Event + UPDATE Snapshot を制御 |

### イベントテーブルの属性設計

| 属性 | 型 | 説明 |
|------|---|------|
| seq_id | integer (PK) | シーケンシャルID |
| {entity}_id | string (FK) | 対象エンティティID |
| event_type | string | イベント種別（状態.tsv の遷移UC に対応） |
| payload | json | イベントペイロード（当時のマスタ値のコピーを含む） |
| occurred_at | datetime | 発生日時 |
| actor_id | string | 操作者ID |

### スナップショットテーブルの属性設計

| 属性 | 型 | 説明 |
|------|---|------|
| {entity}_id | string (PK) | エンティティID |
| current_status | string | 現在のステータス |
| （業務属性） | 各種 | 検索・一覧表示に必要な属性 |
| version | integer | 楽観ロック用バージョン |
| updated_at | datetime | 最終更新日時 |

### トレードオフ

| メリット | デメリット |
|---------|----------|
| 全変更履歴が残る（監査・バグ調査） | 初期実装コスト増（二重書き込み） |
| 「状態」と「事実」の分離でドメインロジックがクリーンに | ストレージ容量増加 |
| イベントログからの状態再構築（Replay）が可能 | イベントスキーマの進化管理が必要 |
