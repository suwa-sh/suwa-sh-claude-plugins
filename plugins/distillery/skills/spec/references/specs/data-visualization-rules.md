# データ可視化設計ルール

データ可視化の設計に適用するルールと原則。Spec 生成時に `_cross-cutting/ux-ui/data-visualization.md` と各 UC の画面仕様でデータ表示が必要な場合に参照する。

## データ可視化の目的

**目的はコミュニケーションである。**
情報を伝達し、対話するために行う。本質的な目的は、相手が認知していない事実を伝え、相手の判断や行動を促し、変えること。

## 基本原則

### 1. 数値情報に意味を与えるのは「比較」

単一の数値だけでは良し悪しの判断ができない。多面的な比較によりストーリーが生まれる:
- **他社比較**: 同業他社との差異
- **過去比較**: 前年度・前月比
- **目標比較**: KPI 目標との差分

### 2. データの種類

**属性データ (Attribute):**
- 名称データ: 人や物の名前（性別、カテゴリ等）
- 地理データ: 座標に紐づくデータ（都道府県、位置情報）
- 日付データ: 年月日（時系列 or 集計項目）
- 順序データ: 順序に意味があるもの（満足度、優先度）

**指標データ (Metric):**
- 間隔尺度: 間隔に意味があるが絶対ゼロがない（気温、スコア）
- 比例尺度: 絶対ゼロが存在する（売上、件数、距離）

### 3. 認知負荷 (Cognitive Load)

- ワーキングメモリの限界: 人間が短期的に保持できる情報は **4-5個**、10秒程度
- 一度に表示する指標・チャートは 4-5 個を上限とする
- ダッシュボードは全体サマリー → ドリルダウンの階層構造にする

### 4. Data-Ink Ratio

チャートから余計な装飾（ノイズ）を削ぎ落とし、データそのもの（シグナル）を強調する:
- 不要なグリッド線、背景色、枠線を排除する
- 3D 効果、影、グラデーションを使わない
- データラベルは必要最小限にする

### 5. ゲシュタルトの法則

人間の知覚特性を活用したグルーピング:
- **近接**: 距離が近いものをグループとみなす → 関連指標は近接配置
- **類同**: 色や形が同じものをグループとみなす → カテゴリ別に色分け
- **連続**: 切れ目のない連続性を見出す → トレンドラインの活用
- **閉合**: 閉じた図形として認識 → カード内にグループ化
- **共通運命**: 同じ動きをするものをグループとみなす → アニメーション連動

## チャート選定ガイド

### 観点別チャート選定

| 観点 | 推奨チャート | 不適切な使用 |
|------|-----------|------------|
| **Comparison（比較）** | 棒グラフ、レーダーチャート | 円グラフで比較 |
| **Composition（構成比）** | 円グラフ（5項目以下）、積み上げ棒グラフ、ツリーマップ | 6項目以上の円グラフ |
| **Relationship（関連性）** | 散布図、バブルチャート | 棒グラフで関連性表現 |
| **Distribution（分布）** | ヒストグラム、箱ひげ図 | 円グラフで分布表現 |
| **Trend（傾向）** | 折れ線グラフ、エリアチャート | 棒グラフで長期トレンド |

### データ表現フレームワーク

| 表現方法 | 用途 | 例 |
|---------|------|---|
| % Change（変化率） | 成長率・鈍化の確認 | 前年比 +15% |
| Variance（差異） | 目標差分の可視化 | 目標比 -5pt |
| Calculated Metric（計算指標） | 割り算指標の活用 | 顧客あたり売上 |
| Added Context（追加コンテキスト） | 比較対象の追加 | 業界平均ラインの追加 |

## 3つの用途別アプローチ

### 1. 自分が気づきを得る（探索的分析）

データと可視化を行き来する探索的アプローチ。仮説の立案・検証に使う。
- チャートの選定 → データの表現方法 → 相関関係の整理

### 2. 他者に伝える（プレゼンテーション）

データストーリーテリングの3要素:
- **ナラティブ**: 物事の本質を明瞭に伝える
- **データ**: 客観的な事実
- **可視化**: データの傾向を把握させる

ステップ: 表現方法選定 → チャート選定 → メッセージに沿った整形 → 不要要素除去 → 重要項目強調 → データ整形 → 信頼感の確保

### 3. 他者が気づきを得る（ダッシュボード）

継続的な運用と利用者の行動変容が目的:
- **後工程からの逆算**: 利用者がデータを見て取るアクションから設計する
- **5W1H の定義**: Who, When, Where, Why, What, How を具体化する
- **情報の階層化**: 全体サマリー → ドリルダウン → 詳細データ

## Spec 記述時の適用

- `_cross-cutting/ux-ui/data-visualization.md`: 上記ルールを適用した全体設計

- design-event.yaml のトークン・コンポーネントとの整合性を保つ
