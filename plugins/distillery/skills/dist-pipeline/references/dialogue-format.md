# 確認推奨項目フォーマット

すべてのスキルで確認対話が発火する際、以下のフォーマットを **必須** とします。

---

## 基本構造

各スキルから確認推奨項目を返却する際は、以下の形式に従う必要があります。

### ❌ NG（旧形式：平文列挙）
```
- 項目1: 推論値
- 項目2: 推論値
- 項目3: 推論値
```

### ✅ OK（新形式：3案+推奨マーク+説明）
```
### <項目ID>: <項目名>
- **Option A** (⭐推奨): <値> — <一行説明（選択理由・効果・代償）>
- **Option B**: <値> — <一行説明>
- **Option C**: <値> — <一行説明>

**推奨理由**: <confidence とその根拠（RDRA/NFR のどこから推論したか）>
```

---

## 詳細ルール

### 1. 項目 ID と名前
- 形式: `### <#番号>: <項目名>`
- 例: `### 1: モデルシステムの規模（全体基準）`

### 2. オプション（最低3案必須）

#### 2-1. オプション行の形式
```
- **Option A** (⭐推奨): <値> — <説明>
```

#### 2-2. 値の形式
- **単一値**: `model2`, `Lv3`
- **範囲**: `〜1,000人`, `〜TB級`
- **説明付き値**: `AWS+RDS (クラウドネイティブ)`

#### 2-3. 説明（一行、30字程度が目安）
- **必須要素**: 「この値を選ぶと何が得られるか、何が失われるか」を明示
- 例:
  - ❌ 「スケーラビリティを確保」（何が？どの程度？）
  - ✅ 「水平スケール対応→ユーザー1万人対応可能、インフラコスト+30%」
  
- 例:
  - ❌ 「標準的な Web サービス」
  - ✅ 「対外接続・個人情報あり→標準的な非機能要求（可用性99%等）」

### 3. 推奨理由

```
**推奨理由**: <confidence: high/medium/low> — <根拠>
```

- **high**: RDRA の複数UC / NFR グレード定義 から直接推論
  - 例: `high — RDRA「利用者数50-500人」+「24h営業」から同時アクセス Lv2 を推奨`
  
- **medium**: RDRA 単一方向情報 / 業界標準 から推論
  - 例: `medium — 図書館システム一般的な可用性要件から Lv2 を推奨`
  
- **low**: ドメイン仮定 / 規模感未確定 から推論
  - 例: `low — 利用者数未確定のため保守的に Lv1 を推奨（規模感プリインタビュー後に再判定推奨）`

## テンプレート（複数項目の場合）

```markdown
## 確認推奨項目

### 1: モデルシステム（全体基準）
- **Option A** (⭐推奨): model2 — 対外接続・重要情報あり。標準的な Web サービス基準で可用性99%等を確保
- **Option B**: model1 — 社内向けのみ。非機能要求を最小化してコスト削減
- **Option C**: model3 — 社会インフラ級。24/365 可用性99.99%。インフラコスト 3〜5倍

**推奨理由**: high — RDRA に「対外向け図書館システム」と記載。個人情報（利用者名）を扱うため model2 推奨

### 2: 災害対策の範囲
- **Option A** (⭐推奨): Lv1 — 同一DC内バックアップのみ。コスト+5%
- **Option B**: Lv0 — 対策不要。被災時は復旧作業で対応
- **Option C**: Lv2 — 遠隔地バックアップあり。RPO数時間。コスト+20%

**推奨理由**: medium — 非機能要求に「災害対策」未記載。業界標準として小規模図書館は Lv1 推奨

...
```

## 使用時の留意点

### A. スキルからの返却時
1. **各スキル SKILL.md** の「確認推奨項目」セクションにおいて、このフォーマットで返却する
2. 不完全な返却（推奨理由なし、オプション2案以下）があれば、**pipeline オーケストレータが再返却を要求**
3. オーケストレータのプロンプトに「このフォーマットに従わない場合は修正リクエストを送付」を追記

### B. オーケストレータ側の検証ロジック
```js
// 擬似コード
if (
  !item.recommendedReason ||  // 推奨理由なし
  item.options.length < 3 ||   // オプション3案未満
  !item.options[0].startsWith("Option A") ||  // 推奨マークなし
  item.description.length < 20  // 説明が短すぎる
) {
  return "このフォーマットに従ってください → (フォーマット仕様へのリンク)"
}
```

### C. プレビュー例（Markdown 表示）

```
### 4: 同時アクセス数（ピーク時）
- **Option A** (⭐推奨): Lv2（〜1,000人） — 図書館規模 500人登録者、ピーク時 50人同時を想定。コスト標準
- **Option B**: Lv1（〜100人） — 小規模・閉域運用の場合。設備コスト大幅削減
- **Option C**: Lv3（〜10,000人） — 自治体域内複数館対応の場合。コスト+50%

**推奨理由**: medium — RDRA に「500人登録者」記載、ピーク同時率 10% と一般的に想定
```

## FAQ

**Q: 「デフォルト値で進める」と言われたら？**
- A: フォーマット通り 3案を返し、「Option A を推奨します、デフォルトで進めてよろしいですか」と確認

**Q: オプション3案が存在しない項目がある場合？**
- A: 「Lv0/Lv1/Lv2 の全3択」のように全選択肢を列挙。2案以下の場合は例外として pipeline に報告

**Q: 数値項目（コスト、RTO秒数）の場合？**
- A: 「範囲値（例：RTO 1-4 時間）+ 説明」で統一。比較可能性を重視

**Q: ユーザーが「個別指定」した場合？**
- A: その値で以降の処理を続行。todo.md には記録しない（ユーザー明示的指定は議論不要）
