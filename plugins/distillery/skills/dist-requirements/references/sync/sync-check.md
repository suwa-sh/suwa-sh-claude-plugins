# 実装同期確認タスク

`src/` の実装と `docs/specs/latest/` の Spec の整合性を確認する。

## 入力

- `docs/specs/latest/` — 最新の Spec
- `src/` — 実装コード（ユーザーのプロジェクトによりパスは異なる場合がある）

## 出力

- 同期レポート（標準出力）
- 不整合がある場合: 補正 Spec イベント（オプション）

## 確認手順

### 1. Spec の読み込み

`docs/specs/latest/` 配下の全 UC の spec.md を読み込み、以下を抽出する:

- UC 名
- 関連 RDRA モデル
- E2E 完了条件（BDD シナリオ）
- ティア別仕様（API パス、画面 URL、データモデルなど）

### 2. 実装の走査

`src/` 配下を走査し、以下の対応関係を確認する:

#### フロントエンド

- tier-frontend.md で定義された画面 URL に対応するコンポーネント/ページが存在するか
- 表示要素が実装されているか
- フォーム項目が仕様と一致しているか

#### バックエンド

- tier-backend.md で定義された API エンドポイントが実装されているか
- リクエスト/レスポンスの型が仕様と一致しているか
- ビジネスルールが実装されているか

#### データモデル

- tier-backend.md で定義されたデータモデルの変更がスキーマに反映されているか

### 3. レポート出力

以下のフォーマットでレポートを出力する:

```markdown
# 実装同期レポート

- 確認日時: {YYYY-MM-DDTHH:MM:SS}
- 対象 Spec: {UC 数} UC
- 対象ソース: {走査したディレクトリ}

## 同期状態サマリ

| 状態 | UC 数 |
|------|-------|
| 同期済み | {数} |
| 未実装 | {数} |
| 仕様不一致 | {数} |
| Spec 不足 | {数} |

## 詳細

### 未実装

| UC名 | 未実装項目 |
|------|-----------|
| {UC名} | {API/画面/データモデルなど} |

### 仕様不一致

| UC名 | 不一致項目 | Spec の定義 | 実装の状態 |
|------|-----------|------------|-----------|
| {UC名} | {項目} | {Spec での定義} | {実装の状態} |

### Spec 不足

実装は存在するが Spec が定義されていない機能:

| 実装 | 種別 | 推奨アクション |
|------|------|--------------|
| {ファイル/クラス/関数} | API/画面/... | Spec の追加を推奨 |
```

### 4. 補正 Spec の生成（オプション）

ユーザーが明示的に指示した場合のみ実行する。

- 仕様不一致がある場合: 実装に合わせた Spec を `docs/specs/events/{event_id}/` に生成
- Spec 不足がある場合: 実装から逆引きした Spec を `docs/specs/events/{event_id}/` に生成
- 生成後、`references/specs/spec-snapshot-update.md` に従い latest を更新

## 注意事項

- `src/` のパスはプロジェクトにより異なる — ユーザーに確認してからパスを決定する
- 実装を変更しない — Spec の確認とレポートのみ行う
- 補正 Spec の自動生成はユーザーの明示的な指示がある場合のみ
- コードの静的解析にとどめ、テストの実行は行わない
