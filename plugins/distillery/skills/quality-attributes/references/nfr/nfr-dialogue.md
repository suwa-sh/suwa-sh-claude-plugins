# Step2: 対話による NFR グレード確認・調整

Step1 の推論結果をユーザーに提示し、選択肢形式の対話で確認・調整するタスク。

## 入力

- Step1 の推論結果サマリ（内部データ）
- `references/nfr-grade-catalog.md` — レベル定義の参照用

## 対話の基本方針

1. **選択肢形式**: すべての確認は番号付き選択肢で提示し、ユーザーは番号で回答できるようにする
2. **必要最小限の対話**: 全238メトリクスを1つずつ確認するのは非現実的。重要かつ確信度が低い項目に絞る
3. **段階的な確認**: モデルシステム → 重要項目 → 要確認項目 の順で進める
4. **推論結果の提示**: 推論根拠を明示し、ユーザーが判断しやすくする
5. **変更の記録**: ユーザーが変更した項目は confidence: "user" に更新する

## 対話フロー

### Phase 0: デプロイ環境の確認

モデルシステム選定の前に、デプロイ環境を確認する。インフラ環境は耐障害性・バックアップ・災害対策のグレードに大きく影響するため、最初に確認する。

```
## デプロイ環境の確認

非機能要求のグレードはインフラ環境によって大きく変わります。

**→ デプロイ先を選択してください:**

1. **AWS**
2. **Azure**
3. **GCP**
4. **オンプレミス**
5. **ハイブリッド（クラウド + オンプレミス）**
6. **未定**
```

ユーザーが 1〜3（クラウド）を選択した場合:
- `nfr-inference-rules.md` の「インフラ環境による補正ルール」の該当テーブルを Step1 推論結果に適用する
- 補正適用後の値を Phase 2 で提示する（補正前→補正後の差分が分かるよう表示）

ユーザーが 4（オンプレミス）を選択した場合:
- 補正なし。モデルシステムデフォルト値をそのまま使用

ユーザーが 5（ハイブリッド）を選択した場合:
- クラウド側の補正テーブルを適用しつつ、オンプレ側の制約を Phase 2 で個別確認

ユーザーが 6（未定）を選択した場合:
- 補正なし。Phase 2 以降はモデルシステムデフォルト値で進め、後から調整可能と案内

### Phase 1: モデルシステムの確認

選択肢形式でモデルシステムを確認する:

```
## モデルシステムの選定

RDRA モデルから、対象システムの社会的影響度を推定しました。

根拠:
- {根拠1}
- {根拠2}

**→ 以下から選択してください:**

1. **モデルシステム1** — 社会的影響がほとんど無い（例: 社内業務システム）
2. **モデルシステム2** — 社会的影響が限定される（例: 中規模サービスシステム） ← 推奨
3. **モデルシステム3** — 社会的影響が極めて大きい（例: 社会インフラシステム）
```

推奨には `← 推奨` マークを付ける。ユーザーは番号（1/2/3）で回答。

### Phase 2: 重要項目の確認

カテゴリごとに推論結果を提示し、変更が必要な項目を選択肢で確認する。

各カテゴリについて以下の形式で提示する:

```
## 重要項目の確認

RDRA モデルから推論した主要な非機能要求グレードです。

### A. 可用性

| # | ID | メトリクス | 推定Lv | 内容 | 根拠 |
|:-:|-----|----------|:------:|------|------|
| 1 | A.1.1.1 | 運用時間（通常） | Lv3 | 1h停止（9時〜翌8時） | BUC「予約管理」が営業時間外利用 |
| 2 | A.1.2.1 | サービス切替時間 | Lv2 | 2h未満（手動切替） | 外部システム2つと連携 |
| 3 | A.4.1.1 | RPO | Lv2 | 数時間前まで | 状態遷移3種 |
| 4 | A.4.1.2 | RTO | Lv2 | 半日以内 | バッチ中心 |

### B. 性能・拡張性

| # | ID | メトリクス | 推定Lv | 内容 | 根拠 |
|:-:|-----|----------|:------:|------|------|
| 5 | B.1.1.1 | 同時アクセス数 | Lv2 | ~1,000 | アクター3種 |
| 6 | B.2.1.1 | レスポンスタイム | Lv3 | 5秒以内 | 一般ユーザー向け画面あり |
| ... | ... | ... | ... | ... | ... |

（C〜F も同様に表示）

**→ 以下から選択してください:**

1. **全項目 OK** — 推論結果をそのまま採用
2. **変更あり** — 変更したい項目の番号とレベルを指定（例: 「#1 → Lv5, #6 → Lv4」）
```

ユーザーが「1」を選択した場合は Phase 3 に進む。
ユーザーが「2」を選択して変更を指定した場合、変更を反映して Phase 3 に進む。

### Phase 3: 要確認項目のヒアリング

推論不能な項目について、選択肢形式でヒアリングする:

```
## 要確認項目

RDRA モデルからは判断が難しい項目です。選択肢から選んでください。

### Q1. 既存システムからの移行はありますか？

1. **新規構築**（既存システムなし）
2. **既存システムからの移行あり（小規模: ~数GB）**
3. **既存システムからの移行あり（中規模: 数十GB〜）**
4. **既存システムからの移行あり（大規模: TB〜）**

### Q2. 運用体制はどのようになりますか？

1. **開発チームが兼務**（営業時間内対応）
2. **専任運用チーム（営業時間内）**
3. **専任運用チーム（24h監視）**

### Q3. セキュリティ規制・コンプライアンス要件はありますか？

1. **特になし**
2. **個人情報保護法対応が必要**
3. **業界固有の規制あり**（金融: FISC、医療: 3省ガイドライン等）
4. **複数の規制に対応が必要**

**→ 各質問の番号をカンマ区切りで回答してください（例: 「1, 2, 2」）**
  不明な項目は「-」で省略できます（例: 「1, -, 2」→ 省略項目はデフォルト適用）
```

ユーザーの回答を反映する。回答があった項目は confidence: "user"、省略された項目は confidence: "default" のまま。

### Phase 4: 最終確認

確定内容のサマリを提示する:

```
## 最終確認

非機能要求グレードが確定しました。

| 項目 | 内容 |
|------|------|
| モデルシステム | {N}（{説明}） |
| 推論確定（自動） | {N}件 |
| ユーザー変更 | {N}件 |
| ヒアリング反映 | {N}件 |
| デフォルト適用 | {N}件 |
| メトリクス合計 | 238件 |

**→ 以下から選択してください:**

1. **OK** — この内容で NFR グレード表を出力する
2. **修正あり** — 修正したい項目を指定する
```

## 出力

対話完了後、確定した NFR グレード情報を Step3 に渡す。
ファイル出力はこのステップでは行わない。

## 対話時の注意事項

- すべての確認は **番号選択式** で行う。自由記述を求めるのは変更理由の補足のみ
- ユーザーが Phase 1 で「1」（全項目OK）等と即答した場合は、Phase 2-3 をスキップして Phase 4 に進む
- Phase 2 の変更指定は `#{行番号} → Lv{N}` 形式を基本とする
- Phase 3 の質問は推論不能項目がある場合のみ表示する。全項目が推論済みならスキップ
- レベルの上げ下げ時は、`nfr-grade-catalog.md` のレベル定義を参照して具体的な内容を併記する
- 対話は最大3ラウンドを目安とし、収束しない場合は現状で確定を提案する
- 差分更新モードでは、変更があった項目のみを Phase 2 で提示する
- Phase 3 の質問項目は RDRA モデルから推論できなかった内容に応じて動的に生成する（上記は代表例）
