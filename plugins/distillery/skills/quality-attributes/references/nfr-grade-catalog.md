# IPA 非機能要求グレード カタログ（2018版）

## 概要

IPA（情報処理推進機構）が策定した「非機能要求グレード」は、非機能要求を体系的に定義するためのフレームワークである。本ドキュメントは LLM エージェントが非機能要求を分析・提案する際のリファレンスとして、全カタログを構造化したものである。

### 構造

```
大項目（6カテゴリ: A〜F）
  └─ 中項目（35サブカテゴリ）
      └─ 小項目（118項目）
          └─ メトリクス（238指標）
              └─ レベル 0〜5
```

### 表記規則

- **★** = 重要項目（重点的に検討すべき項目）
- **M1** = モデルシステム1（社会的影響が極めて限定的）
- **M2** = モデルシステム2（社会的影響が限定的）
- **M3** = モデルシステム3（社会的影響が極めて大きい）
- **●** = 当該モデルシステムの推奨レベル
- レベル定義のないメトリクスはモデルシステムのレベルをデフォルトとする

---

## モデルシステム定義

### M1: 社会的影響が極めて限定的なシステム

- **想定例**: 部門内業務支援、個人向け情報提供
- **特徴**: 停止しても社会的影響は軽微。利用者が限定的
- **方針**: コスト重視。最低限の可用性・セキュリティ

### M2: 社会的影響が限定的なシステム

- **想定例**: 企業基幹業務、中規模ECサイト、自治体サービス
- **特徴**: 停止すると一定の業務影響・経済損失。利用者は数千〜数万
- **方針**: バランス重視。冗長化・セキュリティ対策を適切に実施

### M3: 社会的影響が極めて大きいシステム

- **想定例**: 金融基盤、社会インフラ、大規模公共サービス
- **特徴**: 停止すると社会的混乱・大規模経済損失。利用者は数十万以上
- **方針**: 信頼性・安全性最優先。最高水準の対策を実施

## 活用ガイド（エージェント向け）

### モデルシステム選定の判断基準

1. **M1 を選択**: 社内ツール、部門限定システム、PoC/プロトタイプ
2. **M2 を選択**: 企業基幹業務、顧客向けサービス、中規模ECサイト
3. **M3 を選択**: 金融システム、医療システム、社会インフラ、大規模公共サービス

### 非機能要求定義の進め方

1. **モデルシステムの選定**: 対象システムの社会的影響度からモデルシステムを決定
2. **重要項目（★）の確認**: モデルシステムの推奨レベルをベースラインとして提示
3. **レベル調整**: 業務要件・制約条件に基づきレベルを上下調整
4. **非重要項目の確認**: 必要に応じて非重要項目もレベルを明示
5. **整合性チェック**: カテゴリ間の整合性を確認（例: 可用性を高くするなら運用体制も対応が必要）

### カテゴリ間の主な依存関係

| 起点 | 影響先 | 関係 |
|------|--------|------|
| A.1 継続性（高レベル） | C.5 サポート体制 | 24時間運用なら24時間サポートが必要 |
| A.2 耐障害性（高レベル） | B.3 リソース拡張性 | 冗長構成はリソース拡張にも影響 |
| A.3 災害対策（高レベル） | D.2 移行方式 | DR拠点への移行方式も考慮 |
| A.4 回復性（高RPO/RTO） | C.1.2 バックアップ | リアルタイムレプリケーションが必要 |
| B.1 業務処理量（大規模） | C.1.3 運用監視 | 高度な監視が必要 |
| E.5 認証（高レベル） | C.6 ログ管理 | 監査ログの保管期間も延長 |
| E.7 監査ログ（高レベル） | B.1 データ量 | ログデータ量の増大を考慮 |
