# RDRA → NFR 推論ルール

RDRA モデルの各要素から非機能要求グレードを推論するためのルール定義。

## 推論の基本方針

1. RDRA モデルから読み取れる情報を最大限活用し、可能な限り自動でグレードを推定する
2. 推論には confidence（確信度）を付与し、対話ステップでの確認優先度を決定する
3. 推論不能な項目はモデルシステムのデフォルト値を適用し、confidence: "default" とする

## 入力ファイル

| ファイル | 主な推論先 |
|---------|-----------|
| `docs/rdra/latest/BUC.tsv` | 可用性、性能・拡張性 |
| `docs/rdra/latest/アクター.tsv` | 性能・拡張性、セキュリティ |
| `docs/rdra/latest/外部システム.tsv` | 性能・拡張性、セキュリティ |
| `docs/rdra/latest/情報.tsv` | 運用・保守性、セキュリティ |
| `docs/rdra/latest/状態.tsv` | 可用性（データ整合性） |
| `docs/rdra/latest/条件.tsv` | セキュリティ |
| `docs/rdra/latest/バリエーション.tsv` | セキュリティ |
| `docs/rdra/latest/システム概要.json` | モデルシステム選定、全般 |

## モデルシステム推定ルール

システム概要と BUC の特性からモデルシステムを推定する。

| 判定条件 | 推定 | confidence |
|---------|------|-----------|
| 外部システムに「行政」「医療」「金融」「インフラ」関連が含まれる | model3 | medium |
| BUC に「一般公開」「不特定多数」の業務が含まれる | model2〜3 | medium |
| アクター数が10種以上 or 外部システム数が5以上 | model2 | low |
| 社内業務中心でアクター数5種以下 | model1 | medium |
| 上記いずれにも該当しない | model1 | low |

**注意**: モデルシステムは社会的影響度で決まるため、RDRA だけでは判断が難しい。必ず Step2 で確認する。

## カテゴリ別推論ルール

### A. 可用性

#### A.1 継続性（運用スケジュール）

**推論元**: BUC の業務説明

| BUC の特徴 | 推定グレード | confidence |
|-----------|-------------|-----------|
| 「24時間」「常時」「夜間」を含む業務がある | A.1.1.1 = 4〜5 | high |
| 営業時間内の業務のみ | A.1.1.1 = 1〜2 | high |
| バッチ処理中心 | A.1.1.1 = 1 | medium |
| 外部システムとのリアルタイム連携がある | A.1.1.1 = 3〜4 | medium |

#### A.2 耐障害性

**推論元**: 外部システム数、アクター数

| 条件 | 推定グレード | confidence |
|------|-------------|-----------|
| 外部システムが3つ以上 | A.2.1.1 = 3以上 | medium |
| 外部システムなし | A.2.1.1 = 1〜2 | medium |
| アクターに「一般ユーザー」「顧客」がいる | サーバ冗長化 = 2以上 | low |

#### A.3 災害対策

**推論元**: モデルシステム + BUC の業務特性

| 条件 | 推定グレード | confidence |
|------|-------------|-----------|
| model3 推定 | A.3.1.1 = 3〜4 | medium |
| model2 推定 | A.3.1.1 = 1〜2 | low |
| model1 推定 | A.3.1.1 = 0 | low |

**注意**: 災害対策はビジネス判断が大きいため、confidence は常に low〜medium。

#### A.4 回復性（RPO/RTO）

**推論元**: 状態モデルの複雑さ、BUC の業務特性

| 条件 | 推定グレード | confidence |
|------|-------------|-----------|
| 状態遷移が5種以上の情報がある | RPO = 3以上（直前まで復旧） | medium |
| 金銭・取引に関する状態がある | RPO = 4（データ損失なし） | medium |
| バッチ処理中心 | RPO = 1〜2, RTO = 1〜2 | medium |
| リアルタイム処理中心 | RPO = 3〜4, RTO = 3〜4 | medium |

### B. 性能・拡張性

#### B.1 業務処理量

**推論元**: アクター数、BUC の業務量

| 条件 | 推定グレード | confidence |
|------|-------------|-----------|
| アクター種別 × 想定100ユーザーで同時接続数推定 | B.1.1.1 = 推定値に対応するレベル | low |
| 「大量」「一括」等の記述がある BUC | B.1.1.4（バッチ）= 高め | medium |

#### B.2 性能目標値

**推論元**: BUC の業務特性、アクター

| 条件 | 推定グレード | confidence |
|------|-------------|-----------|
| 一般ユーザー向け画面操作が多い | レスポンス = 3（5秒以内） | medium |
| 管理者向け内部操作中心 | レスポンス = 2（10秒以内） | medium |
| バッチ中心 | バッチ処理時間をデータ量から推定 | low |

#### B.3 リソース拡張性

**推論元**: アクター数の成長見込み、外部システム数

| 条件 | 推定グレード | confidence |
|------|-------------|-----------|
| アクターに「一般ユーザー」がいる | スケールアウト以上（L2+） | low |
| 社内ユーザーのみ | スケールアップ（L1） | medium |

### C. 運用・保守性

#### C.1 通常運用

**推論元**: 可用性の推定結果 + 情報モデル

| 条件 | 推定グレード | confidence |
|------|-------------|-----------|
| 可用性の運用時間に連動 | 運用監視時間 = 可用性と同等 | high |
| 情報モデルの属性数が50以上 | バックアップ方式 = 2以上 | medium |
| 情報モデルの属性数が20未満 | バックアップ方式 = 1 | medium |

#### C.6 ログ管理

**推論元**: セキュリティ要件 + 業務特性

| 条件 | 推定グレード | confidence |
|------|-------------|-----------|
| 金銭・取引関連 BUC がある | ログ保管 = 4（1年）以上 | medium |
| 一般業務 | ログ保管 = 2（3ヶ月） | low |

### D. 移行性

**推論元**: 直接推論困難

推論ルール:
- 基本的にモデルシステムのデフォルト値を適用
- confidence: "default"
- Step2 で必ずヒアリング: 既存システムの有無、移行データ量、並行運用の要否

### E. セキュリティ

#### E.1 前提条件

**推論元**: システム概要

| 条件 | 推定 | confidence |
|------|------|-----------|
| セキュリティポリシーの記載がある | E.1.1.1 = 記載内容に応じて | high |
| 記載なし | モデルシステムデフォルト | default |

#### E.5 アクセス・利用制限

**推論元**: アクター、条件、バリエーション

| 条件 | 推定グレード | confidence |
|------|-------------|-----------|
| アクター種別が3以上 | 認証方式 = 2以上（パスワードポリシー） | medium |
| 条件に「権限」「ロール」がある | アクセス制御 = 2以上 | high |
| 外部システムとの認証連携がある | 認証方式 = 3（多要素認証） | medium |

#### E.6 データ秘匿

**推論元**: 情報モデル

| 条件 | 推定グレード | confidence |
|------|-------------|-----------|
| 「個人情報」「氏名」「住所」等の属性がある | 暗号化（保管）= 1以上 | high |
| 「パスワード」「クレジット」等がある | 暗号化（保管）= 2 | high |
| 外部通信がある | 暗号化（通信）= 1以上 | high |

#### E.7 不正追跡・監視

**推論元**: アクター種別

| 条件 | 推定グレード | confidence |
|------|-------------|-----------|
| 外部ユーザーがアクターにいる | 監査ログ = 2以上 | medium |
| 管理者操作がある | 監査ログ = 2以上 | medium |

### F. システム環境・エコロジー

**推論元**: 直接推論困難（一部を除く）

| 条件 | 推定 | confidence |
|------|------|-----------|
| BUC に「モバイル」「スマートフォン」がある | F.1.1.2 対応ブラウザ = モバイル含む | medium |
| Web アプリケーション想定 | F.1.1.1 対応OS = マルチOS | medium |
| その他 | モデルシステムデフォルト | default |

## インフラ環境による補正ルール

Step2 の対話でデプロイ環境（クラウドプロバイダ / オンプレミス）が判明した場合、以下の補正を適用する。
クラウド環境ではマネージドサービスにより、オンプレミスのデフォルト値より高い水準が標準的に提供される。

### AWS の場合

| ID | メトリクス | オンプレデフォルト | AWS 標準 | 補正根拠 |
|----|----------|-----------------|---------|---------|
| A.2.1.1 | サーバ内の冗長化 | Lv3（N+1手動切替） | **Lv4（N+1自動切替）** | ALB + Auto Scaling による自動フェイルオーバー |
| A.2.3.1 | ネットワーク機器の冗長化 | Lv2（一部冗長化） | **Lv3（全冗長化）** | VPC/ALB のマネージド冗長 |
| A.2.4.1 | 回線の冗長化 | Lv2（一部冗長化） | **Lv3（全冗長化）** | マルチAZ 構成の回線冗長 |
| A.2.5.1 | ストレージの冗長化 | Lv2（RAID5） | **Lv3（ミラーリング以上）** | RDS Multi-AZ 同期レプリケーション、S3 の3AZ以上レプリケーション |
| A.3.1.1 | 災害対策の範囲 | Lv2（コールドスタンバイ） | **Lv3（ウォームスタンバイ）** | Multi-AZ の自動フェイルオーバー（シングルリージョンでもAZ間冗長） |
| C.1.2.1 | バックアップ方式 | Lv2（フル+差分日次） | **Lv3（継続的バックアップ）** | RDS 自動バックアップ＋ポイントインタイムリカバリ（5分粒度） |
| C.1.2.3 | バックアップ世代管理 | Lv2（7世代） | **Lv3（30日以上）** | RDS 最大35日リテンション |

### Azure の場合

| ID | メトリクス | オンプレデフォルト | Azure 標準 | 補正根拠 |
|----|----------|-----------------|-----------|---------|
| A.2.1.1 | サーバ内の冗長化 | Lv3 | **Lv4** | Azure Load Balancer + VMSS による自動フェイルオーバー |
| A.2.3.1 | ネットワーク機器の冗長化 | Lv2 | **Lv3** | Azure VNet/Application Gateway のマネージド冗長 |
| A.2.4.1 | 回線の冗長化 | Lv2 | **Lv3** | Availability Zones 間の回線冗長 |
| A.2.5.1 | ストレージの冗長化 | Lv2 | **Lv3** | Azure SQL Database ゾーン冗長、Blob Storage の LRS/ZRS |
| A.3.1.1 | 災害対策の範囲 | Lv2 | **Lv3** | Availability Zones による自動フェイルオーバー |
| C.1.2.1 | バックアップ方式 | Lv2 | **Lv3** | Azure SQL 自動バックアップ＋ポイントインタイムリストア |
| C.1.2.3 | バックアップ世代管理 | Lv2 | **Lv3** | Azure SQL 最大35日リテンション |

### GCP の場合

| ID | メトリクス | オンプレデフォルト | GCP 標準 | 補正根拠 |
|----|----------|-----------------|---------|---------|
| A.2.1.1 | サーバ内の冗長化 | Lv3 | **Lv4** | Cloud Load Balancing + MIG による自動フェイルオーバー |
| A.2.3.1 | ネットワーク機器の冗長化 | Lv2 | **Lv3** | VPC/Cloud Load Balancing のマネージド冗長 |

| A.3.1.1 | 災害対策の範囲 | Lv2 | **Lv3** | マルチゾーンによる自動フェイルオーバー |
| C.1.2.1 | バックアップ方式 | Lv2 | **Lv3** | Cloud SQL 自動バックアップ＋ポイントインタイムリカバリ |

### 補正の適用ルール

1. Step2 Phase 3 でデプロイ環境が判明した時点で、該当する補正テーブルを適用する
2. 補正後の confidence は `"medium"`（ユーザーが明示的に承認した場合は `"user"`）
3. ユーザーが既に `confidence: "user"` で確定済みの項目には補正を適用しない
4. マルチリージョン構成の場合、A.3.1.1 災害対策は Lv4 以上を検討する
5. オンプレミスの場合、補正は適用しない（モデルシステムデフォルト値をそのまま使用）
